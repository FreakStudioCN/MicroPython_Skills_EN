# Business Code Generation Skill

## Core Philosophy: Unit-Test-Centric Embedded Development

The code generated by this project follows the **unit-test-centric embedded development** methodology:

- **Hardware-Software Decoupling**: Except for `main.py` (the DI assembly entry point), no code file `import`s `machine`. Drivers are injected via factory functions, and task functions receive driver objects as parameters.
- **Dependency Injection & Mocking**: Real drivers and Mock implementations share the same duck-typed interface. Task functions do not care whether they receive real hardware or mock objects. The PC side can directly `import` Mocks to run tests without any hardware.
- **Testability-Driven Design**: Each task function is a pure Python function: given input → produce output, with no global state. Three scenarios — sensor error, missing (None), and normal data — are all covered on the PC side.
- **Full Business Logic Simulation on the Host PC**: Unit tests under `test/pc/` run on CPython and do not depend on the MicroPython runtime. Base libraries under `lib/` (logger, time_helper, etc.) provide CPython fallbacks to support this capability.

---

## Role

Given `project-manifest.json` (phase: scaffold), complete driver download, task code generation, DI assembly, and test generation. **Code understanding and generation are done directly by the LLM, no longer relying on regex scripts.**

### 1A: Line Ending Fix

Source files from upypi / GitHub may contain non-standard line endings like `\r\r\n`, which can cause CPython parsing failures. After downloading, scan `lib/*.py`; if compilation fails, automatically fix:

```bash
cd {project_dir} && python -c "
import os, sys
for fn in os.listdir('firmware/lib'):
    if not fn.endswith('.py'): continue
    path = os.path.join('firmware/lib', fn)
    try:
        with open(path, 'rb') as f: data = f.read()
        compile(data.decode('utf-8'), path, 'exec')
    except (SyntaxError, IndentationError, UnicodeDecodeError):
        # Normalize line endings and remove blank lines between backslash continuations
        text = data.decode('utf-8', errors='ignore')
        text = text.replace('\r\n', '\n').replace('\r', '\n')
        lines = text.splitlines(keepends=True)
        result = []; i = 0
        while i < len(lines):
            line = lines[i]; result.append(line)
            if line.rstrip('\n').endswith('\\\\'):
                i += 1
                while i < len(lines) and lines[i].strip('\n').strip() == '': i += 1
            else: i += 1
        text = ''.join(result)
        with open(path, 'w', encoding='utf-8') as f: f.write(text)
        print(f'[FIX] {fn} line endings normalized')
"

## Phase 2: LLM Understands Driver → Generates Factory + Mock

**For each device in manifest.devices, execute one by one:**

### 2A: Reading Comprehension (The Sole Source for Mock API)

**The Mock's method signatures, parameters, and return types must be extracted directly from the driver source code downloaded to `firmware/lib/` — without relying on any external api_ref strings or pre-built databases.**

Read the following files (if they exist):
```
firmware/lib/<driver>.py          ← Driver source code (authoritative source for Mock API)
firmware/lib/<name>_README.md     ← Usage documentation
firmware/lib/<name>_example.py    ← Example code
```

From the source code, understand:
- **Which is the main driver class**: If there are multiple classes (e.g., `SSD1306` base class + `SSD1306_I2C` subclass), choose the I2C/SPI subclass; if no subclass, choose the first non-Exception class
- **__init__ parameters**: Distinguish I2C address, Pin numbers, configuration parameters
- **Public methods (Mock coverage targets)**: All methods not starting with `_`, record each method's signature (parameter names, default values) and return type. These will directly determine the Mock class's method list and default return values
- **Whether the driver creates its own I2C**: `self.i2c = I2C(...)` → need to modify the driver to support externally passed i2c

example.py can tell you the real calling pattern:
```python
# From example we know: BMP280(i2c=i2c, address=addr), read_compensated_data() returns (temp, press, hum)
```

### 2B: Modify Drivers That Create Their Own I2C (If Needed)

If the driver creates its own I2C in `__init__` (e.g., SHT30's `self.i2c = I2C(scl=Pin(...), sda=Pin(...))`), directly modify `lib/<driver>.py`:

1. Add `, i2c=None` at the end of `def __init__(self, ...)`
2. Wrap `self.i2c = I2C(...)` in `if i2c is not None: self.i2c = i2c else: ...`
3. Keep subsequent assignments like `self.i2c_addr` outside the if/else (needed by both branches)

### 2C: Generate Factory Function → `drivers/<name>_driver/__init__.py`

The factory file is the first link in the DI chain, exporting two functions: `create_<name>` and `scan_<name>_i2c`. Their behavior must be consistent and reliable.

**Reference Source**: The downloaded `lib/<name>_example.py` (i.e., the driver package's `code/main.py`) shows the real I2C address, communication rate configuration, and calling pattern. The LLM must read this file before generating the factory code.

```python
# -*- coding: utf-8 -*-
# @Generated : upy-generate
# @File    : drivers/<name>_driver/__init__.py
# @License : MIT

try:
    from lib.<module> import <DriverClass>
except ImportError:
    <DriverClass> = None

# Default I2C address from driver source / example.py
_<NAME>_DEFAULT_ADDR = <default_addr>


def create_<name>(i2c, address=None):
    """Create <DriverClass> instance. Returns driver object or None on failure."""
    if i2c is None:
        return None
    try:
        obj = <DriverClass>(i2c=i2c, address=address if address is not None else _<NAME>_DEFAULT_ADDR)
        return obj
    except Exception as e:
        from lib.logger import warning
        warning('Driver <NAME> init failed: {}'.format(e))
        print('[DRIVER] <NAME> init failed:', e)
        return None


def scan_<name>_i2c(i2c, address=None):
    """Scan I2C bus for <NAME>. Returns True if device found at address.

    Does NOT create a driver instance — only checks if the I2C address
    responds on the bus. Used by upy-deploy-test for hardware connectivity check.

    NOTE: Some sensors (e.g. GT911) have software-configurable addresses
    that cannot be detected via passive scan. For these, consult the
    downloaded example.py — scan may return False until the device is
    initialized, or require a different detection strategy.
    """
    if i2c is None:
        return False
    addr = address if address is not None else _<NAME>_DEFAULT_ADDR
    return addr in i2c.scan()
```

- I2C devices: Factory accepts `i2c, address=None`, scan function is independent of the driver instance
- GPIO devices (buzzer/led, source=none): Factory accepts `pin_num`, internally creates `Pin(pin_num, Pin.OUT)`. **GPIO devices have no ready-made driver; the LLM must write a reasonable wrapper from scratch** (on/off/toggle/value). The Mock must also cover these methods. GPIO devices do not need a scan function
- SPI devices: Factory accepts `spi, cs_pin`, internally creates `Pin(cs_pin, Pin.OUT)`
- Modified self-creating I2C drivers: Also accept `i2c, address=None`

### 2D: Generate Mock → `drivers/<name>_driver/mock.py`

The Mock's method list and signatures are **extracted directly from the results of Phase 2A's driver source code reading**, covering all public methods of the driver. Default return values match the driver's return types:

```python
# -*- coding: utf-8 -*-
# @Generated : upy-generate
# @File    : drivers/<name>_driver/mock.py
# @License : MIT

class Mock<Name>:
    """Mock <NAME> — same API as real driver, returns canned data."""

    def __init__(self, **kwargs):
        self._<method1> = kwargs.get('<method1>', <default>)
        ...

    def <method1>(self):
        return self._<method1>
    ...
```

Default value rules:
- Returns numeric → `0`
- Returns boolean → `True`
- Returns tuple → `(<typical value>)`, e.g., `(25.0, 60.0)`
- Returns None → do not store `self._xxx`, method body is just `pass`

### 2E: Async Mode Special Handling

If `manifest.scaffold_mode` is `async`:

**Default rule**: The driver itself is not modified (kept synchronous; async conversion is done in the wrapper layer). The factory function returns a synchronous driver object unchanged. The task function form and main.py entry point are determined by the scaffold skeleton phase; the generate phase follows them.

**Exception — blocking methods in the driver must be modified in the driver source code**:

1. **Hardware blocking operations like I2S peripherals**: Consult the MicroPython asyncio official documentation to confirm if there is an async alternative API. If there is (e.g., an async version of `I2S.readinto()`), wrap it in the factory layer; if not, add a non-blocking polling path in the driver.

2. **`time.sleep()` / `time.sleep_ms()` delays**: If the driver has delay waits (e.g., sensor power-on stabilization wait, reset delay), they must be modified:
   - Replace `time.sleep(n)` with `await asyncio.sleep(n)` or `await asyncio.sleep_ms(n)`
   - Change the corresponding method signature to `async def`
   - Reference documentation: https://docs.micropython.org/en/latest/library/asyncio.html

3. **Busy waits like polling for I2C response**: Change to `await asyncio.sleep_ms()` to yield to the event loop.

After modifying the driver, update the corresponding Mock's method signatures (matching `async def`), and use `await` when calling in task coroutines.

## Phase 3: LLM Generates Task Files

Read `manifest.requirements.description` (user requirement description) and `manifest.devices` list → generate task files.

**Constraints (the LLM must satisfy in the generated code):**

1. **Pure functions + DI**: Tasks are pure Python functions, do not `import machine`, all hardware is passed in as driver objects via parameters
2. **Independent exception handling**: Each sensor/device read/write operation has its own try/except; one failing does not affect others
3. **`print()` and `lib.logger` mandatory dual-write**: `install_rotating()` monkey-patches all logger outputs (including `debug()`/`info()`/`warning()`/`error()`) to redirect to `/log/run_*.log` files; the REPL sees nothing. Therefore `print()` is the only visible channel on the REPL, and the following rules must be strictly followed:

   **Scenarios where dual-write (logger + print) is mandatory:**
   - Each successful sensor/device data acquisition → `debug(msg)` + `print(msg)`
   - Display update content → `debug(msg)` + `print(msg)`
   - Initialization/startup/scheduler start → `info(msg)` + `print(msg)`
   - Exceptions/errors → `warning/error(msg)` + `print(msg)`
   - Alarm trigger/recovery → `warning/info(msg)` + `print(msg)`

   **Scenarios where dual-write is NOT used (only one):**
   - I2C scan details, internal driver state changes, etc., not of user concern → only `debug()`
   - Temporary debug prints (to be deleted later) → only `print()`

   Log messages should include a module identifier prefix (e.g., `[sensor]`, `[display]`, `[alarm]`). The specific format is up to the LLM.
4. **Temperature key conflict handling**: When both a temperature/humidity sensor and a barometric pressure sensor are present, the pressure sensor's temperature should be written to `pressure_temp` to avoid overwriting the main temperature
5. **No hardcoded thresholds**: All thresholds and intervals are imported from conf.py
6. **Decorators**: Timer mode uses `@timed_function`, async mode uses `@timed_coro`

**Task log insertion points (the LLM must insert at the corresponding locations; level and specific wording can be adjusted autonomously):**

| Location | Level | Dual-write | Content Requirement |
|----------|-------|:----------:|--------------------|
| Sensor read success | `debug` | **Mandatory** | Include sensor name + actual reading. `debug(msg)` + `print(msg)` both required |
| Sensor read failure | `warning` | **Mandatory** | Include sensor name + exception info. `warning(msg)` + `print(msg)` |
| Alarm triggered | `warning` | **Mandatory** | Include parameter name + current value + threshold. `warning(msg)` + `print(msg)` |
| Alarm recovery | `info` | **Mandatory** | Include parameter name + current value. `info(msg)` + `print(msg)` |
| Display update | `debug` | **Mandatory** | Include summary of current display content. `debug(msg)` + `print(msg)` |
| Display update failure | `warning` | **Mandatory** | Include exception info. `warning(msg)` + `print(msg)` |
| Unexpected Exception | `error`/`exception` | **Mandatory** | Include context (which task, what data being processed). Dual-write logger + print |
| Purely internal state (GC stats, counter increments, etc.) | `debug` | Optional | Internal details not of user concern; logger record is sufficient, print not required |

**LLM decides autonomously:**
- Function signatures (parameter names, order and number of driver objects)
- Data dict key names and structure
- Business logic flow (acquire → process → output)
- How many task files to generate (split by function: sensor acquisition / alarm / display / network, etc.)

## Phase 5: LLM Generates main.py (DI Assembly + Log Initialization)

Read all factory files + task files + conf.py → generate main.py.

**Constraints:**

0. **Startup delay (mandatory first line of code)**: In main.py, after import statements and before any business logic, **must** insert a 3-second startup delay to give mpremote time to re-enumerate USB and establish a connection after a soft reset:

```python
import time; time.sleep(3)  # Boot delay: allow mpremote to reconnect after reset
```

This ensures that the persistent session of deploy/autofix is connected before main.py outputs its first log line, so no startup output is missed.

1. **DI assembly chain**: `machine.I2C/Pin → factory create_xxx() → driver object → task function parameters`, fully connected
2. **I2C address conflict handling**: First try to mount all devices on the same I2C bus; if address conflicts occur, create a second I2C instance
3. **Log must be initialized**:

```python
from lib.logger import install_rotating, getLogger, info, warning, error, setLevel, DEBUG, INFO
from conf import LOG_DIR, LOG_MAX_FILES, LOG_LINES_PER_FILE, LOG_LEVEL

# Install rotating log — all logger output written to /log/run_*.log
# The fmt parameter is decided by the LLM; if not specified, the default "%(levelname)s:%(name)s:%(message)s" is used
install_rotating(LOG_DIR, max_files=LOG_MAX_FILES, lines_per_file=LOG_LINES_PER_FILE)

# Activate conf.LOG_LEVEL — after install_rotating(), logger output only goes to files;
# setLevel() controls which levels are actually written (DEBUG = all, INFO = skip debug)
if LOG_LEVEL == 'DEBUG':
    setLevel(DEBUG)
else:
    setLevel(INFO)
_log = getLogger("main")
```

Add a `print()` to output the firmware name + version to the REPL, then `info(...)` to write to the log.

**Why both print() and rotating log are needed:**
- `print()` → REPL real-time output, immediately visible when mpremote connects
- Rotating log → Written to the device's `/log/run_*.log`, still readable via `mpremote fs` after REPL disconnects for backtracking
- Reference implementation: `G:\MicroPython_Claude_Assistant\device\main.py` (lines 13-24)

4. **I2C scan at startup**: For each I2C device, call `scan_<name>_i2c(i2c)`, dual-write the result with info/warning + print

**main.py log insertion points (the LLM must insert at the corresponding locations; level and specific wording can be adjusted autonomously):**

| Location | Level | Content Requirement |
|----------|-------|---------------------|
| Firmware startup | `info` | Project name + version + board name |
| Rotating log installation | `debug`/`info` | LOG_DIR path |
| I2C bus initialization | `info` | bus id + SCL/SDA pin numbers |
| Each driver creation success | `info` | Driver name + I2C address |
| Each driver creation failure | `warning` | Driver name + exception info |
| I2C scan result | `info`/`warning` | Device name + found/missing |
| Scheduler/event loop start | `info` | tick_ms or equivalent parameter |
| Scheduler main loop exception | `error` | Fatal error, include traceback |

**LLM decides autonomously:** Import organization, initialization order, `install_rotating(fmt=...)` parameters, specific startup log wording, creation method for GPIO/SPI devices.

### 6B: Device-Side Smoke Test `test/device/test_smoke.py`

Only test hardware availability, not business logic.

**Only use assert methods supported by MicroPython unittest** (`assertTrue`, `assertEqual`, `assertIsNotNone`, `assertRaises`). **Prohibit** using `assertIsInstance`, `assertIn`, `assertNotIn`, `assertGreater`, etc., which are CPython extensions.

from machine import I2C, Pin
import unittest
from drivers.<name>_driver import scan_<name>_i2c

class TestSmoke(unittest.TestCase):

    def test_<name>_present(self):
        i2c = I2C(<bus_id>, scl=Pin(<scl>), sda=Pin(<sda>))
        self.assertTrue(scan_<name>_i2c(i2c),

```

**LLM decides autonomously:** Generate a corresponding `test_<name>_present` method for each I2C device, add hardware tests for GPIO/SPI devices. Pin numbers are read from the manifest.

### 7C: MicroPython Import Compatibility Check

Scan all `.py` files under `firmware/` (excluding `firmware/lib/`), extract top-level `import X` and `from X import Y` module names `X`, and check each against a known MicroPython available module whitelist:

**MicroPython built-in + stdlib whitelist (63 items):**

```
sys, os, time, machine, micropython, gc, math, cmath, struct, json,
binascii, collections, errno, hashlib, io, platform, random, re, select,
socket, ssl, array, network, bluetooth, framebuf, uctypes, cryptolib,
deflate, btree, vfs, openamp, lcd160cr, neopixel, esp, esp32, espnow,
rp2, mimxrt, zephyr, wipy, stm, uasyncio, uarray, ubinascii, ucollections,
ucryptolib, uctypes, uerrno, uhashlib, uheapq, uio, ujson, uos, uplatform,
urandom, ure, uselect, usocket, ussl, ustruct, utime, uzlib, threading
```

**LLM execution steps:**

1. Scan the file list (excluding `firmware/lib/` external drivers):

```bash
cd {project_dir} && find firmware -name "*.py" -not -path "firmware/lib/*" | sort
```

2. Read each file, extract top-level `import X` and `from X import Y` module names X (ignore relative imports starting with `.`)

3. For each X, perform a three-level judgment:

```
Is X in the whitelist?
  → Skip (natively supported by MPY)

X is not in the whitelist, but X.py or X/__init__.py exists under firmware/?
  → Error: should be changed to a relative import. Example:
     File firmware/tasks/my_task.py has import logging
     → firmware/lib/logger/logging.py exists
     → Fix to from lib.logger import logging or a relative path import

X is not in the whitelist and does not exist under firmware/?
  → Warning: MicroPython may not have this module. LLM decides if an alternative is needed.

       import typing        → MicroPython does not support it; remove type annotation imports
       import pathlib       → Use os.path / os.mkdir instead
       import dataclasses   → Use regular classes
       import collections   → Only use list/dict (MPY has ucollections but with limited functionality)

```

4. If an auto-fixable import error is found (a corresponding .py file exists within the project) → fix it directly, mark with `[FIX]`

5. If a non-auto-fixable error is found (no alternative module) → print a warning, record it in the review checklist, handle during Phase 8 final review

**This check is performed after flake8/pylint**, because pylint has been configured with an `ignored-modules` whitelist to skip import-error for MPY modules — that is for CPython linting to pass. This check is the reverse: it intercepts cases that pass CPython but would fail on MPY.

### 7D: conf.py Dead Configuration Detection

**This is a critical interception point**: Phase 4 may add constants to `conf.py`, but the task files generated in Phase 3 may not reference them. The result is a configuration item defined but unused — "looks configured, but doesn't actually work."

**LLM execution steps:**

1. Extract all module-level `UPPER_CASE = value` constant names from `conf.py` (excluding Python built-in variables starting with `__`):

```bash
cd {project_dir} && python -c "
import re, sys
with open('firmware/conf.py', 'r', encoding='utf-8') as f:
    text = f.read()
# Match module-level CONSTANT = value
consts = set(re.findall(r'^([A-Z][A-Z_0-9]+)\s*=', text, re.MULTILINE))
print('\n'.join(sorted(consts)))
"
```

2. For each constant, scan all `.py` files under `firmware/` (excluding `firmware/lib/` external drivers and `firmware/conf.py` itself) to check if it is referenced:

```
Reference forms:
  import conf        → subsequent conf.CONSTANT_NAME   ✓
  from conf import X → direct use of X                 ✓
  from conf import (X, Y) → X, Y are used              ✓
```

3. Output the dead configuration list:

```
Dead config (defined in conf.py but not referenced by any firmware/*.py):
  BUZZER_ALARM_DURATION_MS     → No file uses it
  BUZZER_ALARM_INTERVAL_MS     → No file uses it
```

4. For each dead configuration, perform a two-level handling:

```
Is the semantics of the dead config clearly needed?
  ├─ Yes (e.g., BUZZER_ALARM_DURATION_MS means "buzzer should pulse, not beep continuously")
  │   → Fix the task file: introduce this constant in the corresponding logic to make the configuration effective
  │
  └─ No (e.g., a constant invented by the LLM with no practical use)
      → Delete it from conf.py to keep the configuration clean
```

5. After fixing, re-run steps 1-2 to confirm zero dead configurations.

**Typical pitfalls (the LLM must be aware of):**
- Basic constants from the scaffold template (`LOG_DIR`, `SAMPLE_INTERVAL_MS`, etc.) will not be misjudged — they are definitely referenced by main.py or template files
- Constants in `conf.py` only referenced by the `test/` directory are not dead configurations — tests count as usage
- Static configuration values (like I2C addresses, Pin numbers) may only be used in the `main.py` DI assembly phase, not in task files — this is normal

### 8A: Driver API Correctness

### 8B: Requirements Coverage

### 8C: GPIO Device Review

2. Does the Mock cover basic operations like on/off/toggle/value?

### 8D: Test Coverage Review

1. Do `test/pc/test_*.py` cover normal + exception (sensor error) + boundary (sensor is None) scenarios?

## Architectural Constraints (Generated Code Must Follow)

- **Duck-typed DI**: Task functions receive objects; they do not care whether it is a real driver or a Mock
- **Except for main.py, do not import machine**: No .py file directly imports machine.Pin/I2C/SPI/UART
- **Except for lib/, do not use typing generics**: Type annotations only use MPY built-in types
- **Independent try/except**: Each task_tick has independent exception handling; one sensor failing does not affect others
- **Thresholds in conf.py**: Do not hardcode values
- **Mock method signatures match the real driver**: Confirmed by reading the driver source code and example.py
- **All I2C driver factories must export a `scan_<name>_i2c(i2c, address)` function**: Does not depend on a driver instance; only scans the I2C bus to check if the address exists, providing a hardware connectivity judgment basis for subsequent debugging/deployment skills. For devices with software-configurable addresses (e.g., GT911), consult example.py to determine the detection strategy
- **Logs written to the device file system**: main.py installs rotating logging at startup (`lib.logger.install_rotating`). Key process nodes use `print()` + `lib.logger` dual-write (REPL immediately visible + device-side persistence); purely debugging details can use only `print()`. Even if the REPL disconnects, logs can still be read from the device-side files via mpremote

## Strong Constraints

- **Driver source code under lib/ can be modified** (only for: ① adding DI support like i2c=None parameter ② fixing non-standard line endings that affect CPython compilation)
- **Do not generate a global firmware/mock/ directory**
- **In async mode, drivers are not modified by default**, but if a driver contains blocking methods (`time.sleep`, I2S blocking operations, etc.), they must be modified to async versions, and the asyncio official documentation must be consulted to confirm the API; async conversion can also be handled in the wrapper layer
- **Automatically run flake8 + pylint verification + PC tests upon generation completion; do not finish until they pass**
- **Files under lib/ must ensure CPython compatibility**: MPY-specific APIs like `micropython.const`, `time.ticks_ms` must have fallbacks (scaffold files like logger, time_helper may be missing them)
- **Device-side tests only use the MPY unittest assert subset**: `assertTrue`, `assertEqual`, `assertIsNotNone`, `assertRaises`. Prohibit `assertIn`/`assertIsInstance`, etc.
