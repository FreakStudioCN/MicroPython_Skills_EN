# MicroPythonOS App Runtime Test

## Role

Perform MPOS runtime-level smoke testing and lightweight interactive testing on the target App generated by `mpos-gen-app`. By default, only the target App is tested; full OS tests are not run.

Static gates belong to `mpos-gen-app`: `make lint`, manifest, CPython/mpy syntax, MicroPython import, flake8, pylint must be executed immediately after generation/fix. This skill only verifies that these gates are recorded as passed in `generation_result.json`; do not redefine or replace them.

## Unified Project Log

After completing the runtime smoke test or optional Web Port verification and producing `app_test_result.json`, it must be recorded in the project state directory:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-plan-app/scripts/update_plan_state.py record \
  --repo <repo-root> \
  --fullname <fullname> \
  --skill mpos-test-app \
  --phase test-app \
  --result <success|partial|failed|blocked> \
  --artifact app_test_result=<app_test_result.json> \
  --next-skill <handoff.next_skill-or-null> \
  --event "Ran MPOS runtime smoke test"
```

If the failure is in the App itself, `handoff.next_skill` points to `mpos-gen-app` repair; if it is an external OS/tooling blockage, only record blocked/warning, do not let `mpos-gen-app` fix unrelated OS files.

Must use MicroPythonOS repository built-in tools:

- `<repo-root>/scripts/mpos_controller.py`
- `<repo-root>/scripts/run_desktop.sh`
- `<repo-root>/tests/unittest.sh`
- `<repo-root>/internal_filesystem/lib/mpos/ui/testing.py`
- `<repo-root>/internal_filesystem/lib/mpos/testing/`

Do not use `mpos-debug-app` as a prerequisite or reference; device debugging, serial logs, and hardware troubleshooting are outside the default scope of this skill.

Do not directly modify MicroPythonOS OS/build source code to fix the test environment. When encountering missing `_webrepl`, missing desktop binary, missing `libffi-dev`, missing `libv4l-dev`, or other local simulator/tooling issues, by default only mark them as external/tooling blocking. Only when the user explicitly requests fixing the local simulator should this skill's helper be run:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-test-app/scripts/prepare_desktop_tooling.py \
  --repo <repo-root> \
  --build
```

`prepare_desktop_tooling.py` does not edit OS source code; it only creates temporary build shims: uses a fake `pkg-config` pointing to a user-space static `libffi.a`, and when `libv4l2.a` is missing, provides an empty static library for desktop linking only, then calls the existing `<repo-root>/scripts/build_mpos.sh unix`. After execution, the temporary shims are deleted along with the temporary directory; build artifacts are still generated by the original build script.

## Default Tests

By default, run the target App desktop simulator smoke test, divided into two layers:

1. First, must attempt the built-in desktop runner:

```bash
<repo-root>/scripts/run_desktop.sh <fullname>
```

`run_desktop.sh` is the manual/real desktop launch path; it sets `auto_start_app_early` and starts the desktop MPOS. The automation script must execute it with a bounded timeout: if before timeout there is no OS/tooling boot marker, and the output contains `run_desktop.sh: running app <fullname>`, then treat the long-running process as a successful launch. `Error importing mpos.main`, missing `_webrepl`, missing scripts/binaries, etc., are external OS/tooling blockages; a normal traceback/import error after boot only indicates that the desktop runner saw a runtime error, and `mpos_controller.py` must still be used to determine if it belongs to the target App. Before and after probing, restore `prefs/com.micropythonos.settings/config.json` to avoid auto-start configuration polluting subsequent tests.

2. Then run the structured controller smoke test:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-test-app/scripts/run_app_smoke.py \
  --repo <repo-root> \
  --app-fullname <fullname> \
  --generation-result <generation_result.json> \
  --screenshot
```

`run_app_smoke.py` by default includes the above `run_desktop.sh` launch probe, then starts `lvgl_micropy_unix` via `scripts/mpos_controller.py`, first confirms `mpos.main` has booted successfully, then executes in the MPOS runtime:

```python
from mpos.content.app_manager import AppManager
from mpos.ui.testing import wait_for_render
AppManager.start_app("<fullname>")
wait_for_render()
```

Then collects:

- Launch output, returncode, timeout status, and error summary of `run_desktop.sh <fullname>`.
- `AppManager.start_app()` result and traceback.
- Visible text `get_visible_text()`.
- Widget tree summary `get_widget_tree()`.
- Optional screenshot BMP.

If the user provides expected text, add a strong assertion with `--expected-text "<text>"`. Without expected text, the test passes as long as the App can start and the runtime does not throw an exception; empty text is not automatically a failure, because game, Canvas, and image-type Apps may have no label text.

## Optional Tests

### Target Test Files

Only create target App-specific test files when the user requests writing/running test files, or when the generated App explicitly includes testing requirements. Do not run the full `./tests/unittest.sh`.

Available pattern:

```bash
cd <repo-root>
./tests/unittest.sh tests/test_graphical_<app>.py
```

Graphical tests use `mpos.ui.testing.GraphicalTestCase`, `KeyboardTestCase`, `simulate_click`, `wait_for_render`, `capture_screenshot`. Hardware-related logic uses `mpos.testing` mocks. Test files must only cover target App behavior, avoiding changes to OS/framework regression tests.

### Web Port

`https://web.micropythonos.com/` is the online entry point for the MicroPythonOS WebAssembly Web Port. Official documentation states that the Web Port can run the OS in a browser, with `/data` and `/apps` stored in browser IndexedDB, HTTP going through browser `fetch()`, and subject to CORS, no Bluetooth/ADC/IMU/camera, and other browser limitations.

The Web Port is only an optional browser verification, not a default gate:

- Only run when the user explicitly requests Web Port verification; add `--web-port-check` to `run_app_smoke.py`.
- The latest local code already includes `scripts/run_web.sh` and `scripts/web_port/`; if missing, record as `skipped_missing_local_web_tooling`.
- `run_app_smoke.py --web-port-check` only calls `scripts/run_web.sh --no-build` to serve existing `web/` artifacts, and checks `http://127.0.0.1:<port>/` with an HTTP GET.
- If `web/index.html`, `web/micropython.js`, `web/micropython.wasm`, or `web/micropython.data` do not exist, record as `skipped_missing_web_artifacts`; do not automatically rebuild unless the user requests otherwise.
- `scripts/run_web.sh` by default first executes `scripts/build_mpos.sh web`, requiring Emscripten `emcc` or an auto-sourceable `../emsdk`/`../../emsdk`.
- If `emcc` is unavailable, record as `skipped_missing_emscripten`, do not treat it as an App failure.
- Only add `--web-port-browser-check` when the user explicitly requests browser automation; missing Chrome/Chromium, headless launch failure, or timeout only record skipped/warning, do not fail the target App.
- Do not use the Web Port to replace `scripts/mpos_controller.py` desktop smoke.

The latest code is consistent with the official Web Port documentation; the local commands are:

```bash
scripts/build_mpos.sh web
scripts/run_web.sh
```

Optional smoke command:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-test-app/scripts/run_app_smoke.py \
  --repo <repo-root> \
  --app-fullname <fullname> \
  --web-port-check
```

`scripts/build_mpos.sh web` stages `internal_filesystem` to `web/.preload_internal_filesystem`, injects web-only `_thread`, `socket`, `aiorepl`, `_webrepl`, `websocket`, `aiohttp`, `machine.Timer` shims, and outputs `web/micropython.{html,js,wasm,data}` and `web/index.html`. This verifies browser/WASM-side compatibility; the default App smoke test still prioritizes the local desktop simulator.

## Failure Handling

On failure, output a structured result and hand back to `mpos-gen-app repair`, including:

- Failure phase: `generation_result_static_gates`, `desktop_runner_launch`, `desktop_boot`, `app_start`, `expected_text`, `screenshot`.
- command, cwd, returncode or exception type.
- Key excerpts from stdout/stderr/traceback.
- Visible text and widget tree summary.
- Screenshot path (if generated successfully).
- Involved App files and whether repair is allowed.

Only hand App self-start/render/interaction failures to `mpos-gen-app repair`. If the failure is due to missing `lvgl_micropy_unix`, `mpos.main` boot failure, missing `_webrepl`, missing local web tooling, missing Emscripten, missing web build artifacts, full OS test failure, hardware unavailability, or external service unreachability, mark as external blockage or warning, do not let `mpos-gen-app` modify unrelated OS files, and do not directly modify OS source code.

## Output JSON

Use `templates/app_test_result.json` as the field template, and validate with the script:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-test-app/scripts/validate_app_test_result.py \
  /path/to/app_test_result.json
```

On success:

- `schema_version` is `mpos-test-app-v1`.
- `phase` is `test-app`.
- `app.fullname` points to the target App.
- `checks[]` contains at least `generation_result_static_gates`, `desktop_runner_launch`, and `desktop_smoke`.
- If `--web-port-check` was run, `checks[]` appends `web_port`, and `required` must be `false`.
- `handoff.next_skill` is `null`, or `"mpos-gen-app"` if the user wants to continue with repair.
