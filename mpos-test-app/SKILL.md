# MicroPythonOS App Runtime Test

## Role

Perform MPOS runtime-level smoke testing and lightweight interactive testing on the target App generated by `mpos-gen-app`. By default, only the target App is tested; full OS tests are not run.

Static gates belong to `mpos-gen-app`: `make lint`, manifest, CPython/mpy syntax, MicroPython import, API cross-validation, flake8, pylint, and App-only change checks must be executed immediately after generation/repair. This skill only verifies that these gates are recorded as passed in `generation_result.json`; do not redefine or replace them.

## User-Visible Language

Follow the language continuity rule of `mpos-dev`: if the current workflow starts in Chinese, continue with Chinese for test summaries, failure attribution, and next-step suggestions; if it starts in English, continue with English. Code, commands, paths, API names, and JSON field names remain in English.

## Default Test

By default, run the target App desktop simulator smoke test in two layers:

1. First, must attempt the built-in desktop runner:

```bash
<repo-root>/scripts/run_desktop.sh <fullname>
```

`run_desktop.sh` is the manual/real desktop launch path; it sets `auto_start_app_early` and starts the desktop MPOS. The automation script must execute it with a bounded timeout: if no OS/tooling boot marker appears before the timeout, and the output contains `run_desktop.sh: running app <fullname>`, treat the long-running process as a successful launch. `Error importing mpos.main`, missing `_webrepl`, missing script/binary, etc., are OS/tooling external blockages; ordinary traceback/import errors after boot only indicate that the desktop runner saw a runtime error, and you must continue using `mpos_controller.py` to determine if it belongs to the target App. Before and after probing, restore `prefs/com.micropythonos.settings/config.json` to avoid auto-start configuration polluting subsequent tests.

2. Then run the structured controller smoke:

```bash
PYTHONDONTWRITEBYTECODE=1 /home/leeqingshui/mp_env/bin/python \
  /home/leeqingshui/MicroPython_Skills/mpos-test-app/scripts/run_app_smoke.py \
  --repo <repo-root> \
  --app-fullname <fullname> \
  --generation-result <generation_result.json> \
  --screenshot
```

`run_app_smoke.py` by default includes the above `run_desktop.sh` launch probe, then starts `lvgl_micropy_unix` via `scripts/mpos_controller.py`, first confirms `mpos.main` has booted successfully, then executes in the MPOS runtime:

```python
from mpos.content.app_manager import AppManager
from mpos.ui.testing import wait_for_render
AppManager.start_app("<fullname>")
wait_for_render()
```

Then collects:

- Launch output, returncode, timeout status, and error summary of `run_desktop.sh <fullname>`.
- `AppManager.start_app()` result and traceback.
- Visible text `get_visible_text()`.
- Widget tree summary `get_widget_tree()`.
- Optional screenshot: keep the BMP raw artifact, and by default convert to a PNG publish-ready artifact.

If the user provides expected text, use `--expected-text "<text>"` to add a strong assertion. Without expected text, passing requires only that the App can start and the runtime does not throw an exception; empty text should not automatically be considered a failure, because game, Canvas, and image-based Apps may have no label text.

## Optional Tests

### Target Test File

Only create a dedicated test file for the target App when the user asks to write/run a test file, or when the generated App explicitly includes testing requirements. Do not run the full `./tests/unittest.sh`.

Available pattern:

```bash
cd <repo-root>
./tests/unittest.sh tests/test_graphical_<app>.py
```

Graphical tests use `mpos.ui.testing.GraphicalTestCase`, `KeyboardTestCase`, `simulate_click`, `wait_for_render`, `capture_screenshot`. Hardware-related logic uses `mpos.testing` mocks. The test file must only cover the target App's behavior, avoiding changes to OS/framework regression tests.

### Web Port

`https://web.micropythonos.com/` is the online entry point for the MicroPythonOS WebAssembly Web Port. Official documentation states that the Web Port can run the OS in a browser, `/data` and `/apps` exist in browser IndexedDB, HTTP goes through browser `fetch()`, and is subject to CORS, no Bluetooth/ADC/IMU/camera, and other browser limitations.

The Web Port is only an optional browser verification, not a default gate:

- Only run when the user explicitly requests Web Port verification; add `--web-port-check` to `run_app_smoke.py`.
- The current latest local code already includes `scripts/run_web.sh` and `scripts/web_port/`; if missing, record as `skipped_missing_local_web_tooling`.
- `run_app_smoke.py --web-port-check` only calls `scripts/run_web.sh --no-build` to serve existing `web/` artifacts, and checks `http://127.0.0.1:<port>/` via HTTP GET.
- If `web/index.html`, `web/micropython.js`, `web/micropython.wasm`, or `web/micropython.data` do not exist, record as `skipped_missing_web_artifacts`; do not auto-rebuild unless the user requests otherwise.
- `scripts/run_web.sh` by default first executes `scripts/build_mpos.sh web`, which requires Emscripten `emcc` or an auto-sourceable `../emsdk`/`../../emsdk`.
- If `emcc` is unavailable, record as `skipped_missing_emscripten`, do not treat it as an App failure.
- If the Web build/link reports errors like `machine_timer_type` symbol or toolchain errors, classify them as OS/Web port tooling issues; first suggest installing or fixing the Web build dependencies/toolchain, do not let `mpos-gen-app` modify the target App, and do not misclassify it as a missing ordinary Python dependency.
- Only add `--web-port-browser-check` when the user explicitly requests browser automation; missing Chrome/Chromium, headless launch failure, or timeout should only be recorded as skipped/warning, not cause the target App to fail.
- Do not use the Web Port as a substitute for `scripts/mpos_controller.py` desktop smoke.

The current latest code is consistent with the official Web Port documentation; the local commands are:

```bash
scripts/build_mpos.sh web
scripts/run_web.sh
```

`scripts/build_mpos.sh web` stages `internal_filesystem` to `web/.preload_internal_filesystem`, injects web-only `_thread`, `socket`, `aiorepl`, `_webrepl`, `websocket`, `aiohttp`, `machine.Timer` shims, and outputs `web/micropython.{html,js,wasm,data}` and `web/index.html`. This verifies browser/WASM-side compatibility; the default App smoke test still prioritises the local desktop simulator.

## Failure Handling

On failure, output a structured result and hand back to `mpos-gen-app repair`, including:

- Failure phase: `generation_result_static_gates`, `desktop_runner_launch`, `desktop_boot`, `app_start`, `expected_text`, `screenshot`.
- command, cwd, returncode or exception type.
- Key snippets of stdout/stderr/traceback.
- Visible text and widget tree summary.
- Screenshot path (if generated successfully).
- `manual_preview_commands`: release ELF fast desktop command, local build desktop command, optional Web Port command.
- Involved App files and whether repair is allowed.

Only hand App startup/render/interaction failures to `mpos-gen-app repair`. If the failure is due to missing `lvgl_micropy_unix`, `mpos.main` boot failure, missing `_webrepl`, missing local web tooling, missing Emscripten, missing web build artifacts, full OS test failure, hardware unavailability, or external service unreachability, mark as external blockage or warning, do not let `mpos-gen-app` modify unrelated OS files, and do not directly modify OS source code.
