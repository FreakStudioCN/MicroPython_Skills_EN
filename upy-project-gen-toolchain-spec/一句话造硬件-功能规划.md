# One-Sentence Hardware — AI Embedded Generation Tool Feature Planning

> Goal: Users only need to describe their requirements in natural language, and the system automatically completes the entire process from selection, code generation, compilation, to flashing.

---

## Core User Journey

```
User says one sentence
    ↓
AI understands intent + hardware selection
    ↓
Automatically generates code + wiring diagram
    ↓
Library dependency auto-resolution and installation
    ↓
One-click compilation + flashing
    ↓
Serial debugging + automatic error repair
```

## Overall Architecture

### Two Skill Systems

All capabilities of this project are organized in the form of **Skills (SKILL.md)**, stored in `G:\MicroPython_Skills\`, totaling 25, divided into two groups:

```
G:\MicroPython_Skills\
├── One-Sentence Hardware Pipeline (10)          ← Detailed description in this file
│   ├── upy-analyze/         Phase 1  Requirement parsing + driver search
│   ├── upy-select-hw/       Phase 2  MCU selection + pin assignment + BOM
│   ├── upy-scaffold/        Phase 3  Project skeleton generation
│   ├── upy-generate/        Phase 4  Business code generation
│   ├── upy-simulate/        Phase 4.5 PC-side full-process simulation
│   ├── upy-deploy/          Phase 5  One-click flashing and running
│   ├── upy-autofix/         Phase 6  Orchestration coordination layer (hierarchical decision-making + delegated repair)
│   ├── upy-wiring/          Phase 7a Wiring diagram generation
│   ├── upy-diagram/         Phase 7b Architecture diagram + flowchart
│   └── upy-gen-driver/      Exception path  Uncommon hardware driver generation
│
├── Driver Development Standardization (15)              ← See README.md for details
│   ├── Standardization: upy-norm-driver, upy-norm-main, upy-norm-pkg
│   ├── Generation: upy-gen-main, upy-gen-readme, upy-gen-pkg
│   ├── Optimization: upy-opt-driver, upy-slim-driver
│   ├── Packaging: upy-pack-driver
│   ├── Deployment: upy-deploy-test
│   ├── Project: upy-project
│   ├── Tools: upy-pkg-guide, fetch-doc
│   └── Devices: mpremote-device-interaction, mpremote-file-transfer, mpremote-live-session
│
└── upy-project-gen-toolchain-spec/    ← Directory of this file
    ├── 一句话造硬件-功能规划.md        Overall architecture + process + module design
    ├── project-manifest.schema.json   Data contract for each phase
    ├── wiring.schema.json             Data contract for wiring diagram
    └── diagram.schema.json            Data contract for architecture diagram
```

### Skill Design Principles

**Each Skill = SKILL.md (LLM instruction) + scripts/ (deterministic scripts)**:

| Layer | Responsibility | Example |
|-------|---------------|---------|
| SKILL.md | LLM role positioning + execution steps + decision rules + constraints | Natural language requirement → component list |
| scripts/*.py | Deterministic operations (data collection, format conversion, rendering) | triage.py collects logs + I2C scan |

**Skills pass state through project-manifest.json**, each Phase only writes its own fields (see schema).

### LLM vs Script Division of Labor Principle

```
Deterministic, reproducible, no judgment required → scripts/*.py
    ├── Log parsing (regex matching)
    ├── I2C hardware scan (mpremote exec)
    ├── Data format conversion (JSON → Markdown → Mermaid)
    ├── File rendering (SVG/PNG generation)
    └── Git snapshot/rollback

Requires judgment, context understanding, creativity → LLM (guided by SKILL.md)
    ├── Requirement intent decomposition
    ├── MCU selection decision
    ├── Code generation
    ├── Hierarchical decision-making (which upstream skill autofix delegates to)
    └── Scenario design (data generator for simulate)
```

### Data Flow

```
User natural language
    ↓
[Phase 1] upy-analyze ──→ project-manifest.json (phase: analyze)
    ↓
[Phase 2] upy-select-hw ──→ project-manifest.json (phase: select-hw, +mcu, +pinout, +bom)
    ↓
[Phase 3] upy-scaffold ──→ firmware/ skeleton + project-manifest.json (phase: scaffold)
    ↓
[Phase 4] upy-generate ──→ firmware/ complete code + project-manifest.json (phase: generate)
    ↓
[Phase 4.5] upy-simulate ──→ test/pc/sim_main.py (CLI+rich, PC verification)
    ↓
[Phase 5] upy-deploy ──→ Device runs + output determination
    ↓                    ↓ FAIL
    PASS                  [Phase 6] upy-autofix ──→ Hierarchical decision → delegate to upstream skill → re-deploy
    ↓
[Phase 7] upy-wiring + upy-diagram (can run in parallel)
```

## Feature Modules

### 1. Natural Language Understanding (Input Layer)
- User describes project requirements in one sentence (supports both Chinese and English)
- AI decomposes intent: functional requirements / sensors / communication methods / trigger conditions
- Supports follow-up clarification (multi-turn dialogue to complete information)
- Example: `"Make a temperature and humidity monitor that sends a WeChat notification when the threshold is exceeded"`

### 2. Hardware Selection Recommendation
- Automatically recommends development board models based on requirements (ESP32 / Arduino Uno / STM32, etc.)
- Lists required sensor/module list (including model, quantity)
- Provides purchase suggestions (price range, alternatives)
- Considers user's existing hardware (can inform about owned devices)

### 3. Wiring Diagram Generation
- Automatically generates pin connection diagram (which pin connects where)
- Visual wiring schematic (Fritzing style or text table)
- Annotates precautions (voltage, current limits, etc.)

### 4. Code Generation
- Supports multiple languages: Arduino C++ / MicroPython / ESP-IDF
- Generates complete runnable project (including main file + configuration file)
- Code includes necessary comments, suitable for beginners
- Supports Blockly blocks and code bidirectional synchronization (optional)

### 5. Library Dependency Auto-Management (Key Difficulty)
- Automatically identifies third-party libraries required by the code
- Automatically downloads and installs correct versions (avoids version conflicts)
- Project-level dependency isolation (different projects do not interfere with each other)
- Reference: AutoEmbed's component-aware library resolution

### 6. Compilation and Flashing
- Built-in compiler or cloud compilation (shortens local environment setup time)
- Automatically detects serial / COM port
- One-click flashing to device
- Compilation errors automatically AI-explained + repair suggestions

### 7. Serial Debugging and Feedback
- Real-time serial monitor (log output)
- AI interprets device output (automatic alarm for anomalies)
- Automatic code iteration repair loop (generate → test → repair)

### 8. AI × Git Version Control

- Automatic `git init` upon project creation, no manual user operation required
- Automatic commit after each AI code generation/modification, generating standardized descriptions:
  - `feat: Add I2S microphone driver`
  - `fix: Fix screen display BUG`
  - `refactor: Optimize WiFi reconnection logic`
- After manual code modification, AI automatically identifies changes and generates commit description
- One-click rollback to the last working version upon flashing failure / device anomaly
- Hides Git complexity from users, displays version history as a **timeline UI** (similar to Figma version history)
- Version history = project development log, easy to share, reproduce, and collaborate

> Differentiation: No competitor has this feature; it's a direct solution to the pain point of "can't go back after messing up" for beginner makers.

### 9. Project Template Library
- Common scenario templates (Smart Home / Environmental Monitoring / Robotics / Displays / Network Communication)
- One-click reuse + custom modification
- Community-shared templates

## User Entry Strategy

**First Entry: A single input box on the landing page**
The product opens with `"What do you want to make?"` — no board selection, no registration, just state the requirement directly. Zero-friction cold start, similar to ChatGPT experience.

**Second Entry: Web project library**
The project library webpage is naturally an SEO entry point. User searches "ESP32 temperature humidity" → landing page shows project → guides to "Open in IDE" → triggers download/installation.

**Third Entry: VS Code Plugin (Lightweight Hook)**
Does not implement full functionality, only AI code generation + jump to desktop IDE for flashing. Covers existing developers, lowers migration cost.

## VS Code Plugin Architecture Detailed Design

### Overall Architecture

```
User Operation (VS Code Plugin)
    │
    ├── Local Execution (Plugin Direct Call)
    │     ├── mpremote → Serial flash/debug (physical serial port, server cannot reach)
    │     ├── git      → Version control (operates local repository)
    │     └── File I/O → Project files (VS Code workspace)
    │
    └── Server API Call (Everything else)
          ├── /analyze     → Phase 1 Requirement parsing + upypi/GitHub search (LLM)
          ├── /select-hw   → Phase 2 Hardware selection recommendation (pin mapping maintained on server)
          ├── /scaffold    → Phase 3 Project skeleton generation (template filling)
          ├── /generate    → Phase 4 Business code generation + syntax check + test generation (LLM, internalized original Phase 5/6)
          ├── /fix-error   → Phase 6 Error repair analysis (LLM)
          ├── /gen-driver → Exception path PDF/Arduino → MPY driver conversion
          └── /git-msg     → Commit description generation (diff → LLM)
```

**Benefit of placing pin mapping on the server**: New board support requires no plugin update from the user; the server update takes effect immediately.

### VS Code Plugin Technical Feasibility

The plugin is essentially a Node.js process; all the following capabilities are supported:

| Capability | API |
|------------|-----|
| Call local mpremote / git | `child_process.spawn` |
| Call server API | `fetch` / `axios` |
| Render wiring diagram, timeline, chat interface | `WebviewPanel` |
| Read/write local project files | `vscode.workspace.fs` |
| Display mpremote real-time output | `vscode.window.createTerminal` |

**The plugin is just a local execution layer + a shell for calling server APIs, lightweight, with all business logic on the server.**

### Module 1: Requirement Analysis + Hardware Selection (Phase 1-2)

**Phase 1 — upy-analyze: Natural Language → Component List + Driver Search**

```
User one sentence
  → LLM decomposes intent: functional requirements / sensors / communication methods / trigger conditions
  → Multi-keyword parallel search of upypi API (3 keyword variants per component)
  → Found on upypi → Read package.json + README.md → Extract API reference
  → Not found on upypi → GitHub Search API fallback → If Arduino code found, mark it
  → No driver found at all → Trigger upy-gen-driver cold hardware path
```

Output: Component list JSON + Driver package list + API usage reference

**Phase 2 — upy-select-hw: Hardware Selection + Pin Assignment**

```
Component list + User-specified MCU (optional)
  → MCU recommendation rules: WiFi → ESP32 / Low power → ESP32-C3 / Simple GPIO → RP2040
  → Built-in pin database (ESP32/ESP32-S3/ESP32-C3/Pico/ESP8266)
  → I2C address conflict detection + shared bus device grouping
  → Output: BOM table (model/quantity/unit price/notes) + Pin assignment table
```

**Execution Location:** LLM call, upypi/GitHub search → Server; Pin database can be locally cached

---

### Module 2: Project Scaffolding + Business Code Generation (Phase 3-4)

**Phase 3 — upy-scaffold: Complete Directory Skeleton Generation**

```
MCU model + Pin assignment table + Scheduling type (Timer/asyncio/while-loop)
  → Generate firmware/ complete skeleton:
    firmware/board.py              # Pin mapping (no hardware instantiation)
    firmware/conf.py               # Static configuration constants
    firmware/libs/
      scheduler.py                 # Timer task scheduler (copied from Scheduler.py)
      rotating_logger.py           # MPY/CPython dual-platform rotating log
      time_helper.py               # timed_function timing decorator
    firmware/tasks/
      maintenance.py               # GC check + error callback
    firmware/main.py               # Scheduling entry skeleton
  → Generate PC-side scripts:
    tools/read_device_log.py       # mpremote reads device rotating log
    tools/log_report.py            # Log parsing → JSON test report
```

**Phase 4 — upy-generate: Business Code Generation + Syntax Check**

```
Project skeleton + Driver API reference
  → Download/standardize driver (call upy-norm-driver)
  → Generate firmware/tasks/sensor_task.py (tick() pattern + deque filtering)
  → Fill firmware/main.py (init → register tasks → start scheduler)
  → black --line-length 100 + flake8 static check
```

**Execution Location:** Code generation, driver standardization → Server; File writing → Local

### Module 3: Syntax Check + Test Generation (Internalized into Phase 4 upy-generate)

**Original Phase 5 upy-verify-docs → Downgraded to CI/CD Background Task:**

```
CI/CD Scheduled Task
  → scripts/extract_mpy_calls.py Extract MPY-specific API changes
  → Verify function signatures/parameters/return values against MicroPython official documentation
  → Common errors: I2C constructor signature, time.sleep vs sleep_ms, json vs ujson
  → Output change summary → Feedback to upy-generate SKILL.md reference document links
```

No longer an interactive skill, but CI/CD automation: MicroPython submodule update → Extract API changes → Update skill.md reference documentation.

**Original Phase 6 upy-inject-test → Cancelled:**

The following features have been directly internalized by upy-generate during the code generation phase:
- Debug points + try/except → Phase 3 Constraint #2 (Independent exception handling)
- @timed_function decorator → Phase 3 Constraint #6
- I2C scan function → Phase 2C `scan_<name>_i2c`
- PC unittest generation → Phase 6A (Normal/Missing/Exception three scenarios)
- Device smoke test → Phase 6B (MPY unittest assert subset)
- Log injection → Phase 3/5 Log insertion point table

Internalization is better than post-injection: generate correct code directly by constraints during generation, rather than "generate first → then script modify".

### Module 4: Visualization Generation (Phase 7, Can Run in Parallel)

**Phase 7a — upy-wiring: Wiring Diagram Generation**

```
LLM reads all .py source code in firmware/ → Extracts pins/addresses/buses → Cross-validates with manifest
  → LLM generates docs/wiring.json (conforms to wiring.schema.json)
  → scripts/validate_json.py validates intermediate JSON
  → scripts/render_wiring_local.py renders Mermaid wiring diagram
  → Output docs/wiring.md + .svg + .png + .html (self-contained browser page, Mermaid.js CDN)
    + docs/wiring_pins.md (Pin cross-reference table)
  → MCU box + Component box + Wires (color-coded for I2C/SPI/UART/GPIO) + Warning annotations
```

**Phase 7b — upy-diagram: Software Architecture Diagram + Flowchart + Data Flow Diagram**

```
LLM reads all code in firmware/ + manifest → Analyzes code structure/execution flow/data flow
  → LLM generates docs/diagram.json (conforms to diagram.schema.json, supports simple/medium/detailed three complexity levels)
  → scripts/validate_json.py validates intermediate JSON
  → scripts/render_diagram_local.py renders 3 types of Mermaid diagrams
  → Output each diagram .md + .svg + .png + .html (self-contained browser page, Tab switching charts/source code, dark mode adaptive)
    docs/architecture.md — Layered architecture diagram (graph TB)
    docs/flowchart.md   — Execution flowchart (sequenceDiagram)
    docs/data_flow.md   — Data flow diagram (graph LR)
```

**Execution Location:** LLM generates JSON → Server; Script validation + Mermaid rendering → Local

### Module 5: Deployment and Running (Phase 5)

**Phase 5 — upy-deploy: mpremote Upload + Run**

Reuses existing `mpremote-device-interaction`, `mpremote-file-transfer`, `mpremote-live-session` three skills:

```
Confirm COM port (mpremote connect list)
  → Create remote directories (/lib, /drivers, /log)
  → Recursively upload all files (firmware/ → device)
  → Verify file integrity (remote ls compared to local file list)
  → soft-reset → Run main.py
  → Capture output (max 120s), no analysis, pass to next phase
```

**Execution Location:** All local (mpremote serial operations)

### Module 5-B: PC-Side Full-Process Simulation (Phase 4.5)

**Phase 4.5 — upy-simulate: Business Logic Verification Without Hardware**

> This is a code generation skill, not a runtime framework. After LLM reads all code in firmware/, it autonomously designs the scheduling scheme, Mock data generator, and visualization form, generating `test/pc/sim_main.py`.

```
Step 1: LLM fully reads firmware/ context
  → main.py (callback wrapping + task registration + data flow)
  → tasks/*.py (function signatures + business logic branches)
  → drivers/*/mock.py (Mock __init__ parameters + _raise_on injection points)
  → conf.py (thresholds + sampling rate + refresh interval)
  → lib/scheduler/*.py (Scheduler API)
  → project-manifest.json (mode: timer|async|thread)

Step 2: LLM autonomously designs 4 decisions
  Scheduling scheme: timer → time.sleep loop | async → asyncio.run | thread → threading
  Mock data generator: gen_xxx(tick) function, data changes over time (core—static constants prohibited)
  Visualization: CLI+rich first (default for all projects) | tkinter GUI optional experimental mode
  Data scenarios: normal / temp_rising / temp_dropping / intermittent_failure / sensor_death

Step 3: Generate test/pc/sim_main.py (+ sim_scheduler.py in timer mode)
  → flake8 + pylint validation → Loop repair until pass
  → AskUserQuestion whether to run

Step 4: Run + Determination
  → python test/pc/sim_main.py --ticks 30 --mode gui --scenario temp_rising
  → Observe: Does the alarm trigger when temperature crosses the threshold? Does the sensor recover after failure?
  → PASS → Enter deploy
  → FAIL → Fix sim_main.py (bugs in the simulation script itself), do not modify firmware/
```

**Core Constraints:**
- **Do not modify any files under firmware/** (unless a definite bug is found)
- Mock data must use generator function `gen_xxx(tick)`, static constants prohibited
- Projects with display devices use rich Panel to simulate virtual screen
- Can be called by upy-autofix: After repair, first PC verify (2-3s), then decide whether to flash

**Execution Location:** All local (CPython runtime, no hardware dependency)

### Module 6: Automated Debugging Closed Loop (Phase 6)

**Phase 6 — upy-autofix: Orchestration Coordination Layer (Not an Independent Repair Engine)**

Core positioning: autofix does not write repair code itself, nor does it execute deployment itself. It is a lightweight orchestration layer—parse error → hierarchical decision → delegate to upstream skill for repair → verify.

```
Step 1: Read logs (autofix's unique capability)
  → python tools/read_device_log.py --port {COM}
  → Device-side rotating_logger writes to /log/run_*.log → PC-side script reads and merges

Step 2: Parse error + Hardware detection (autofix's unique capability)
  → Parse priority P0→P3:
    P0: Traceback / __MPY_ERR__: (Python runtime exception)
    P1: rst cause:4 / Guru Meditation Error (Hardware reset/kernel panic)
    P2: [FAIL] xxx / MemoryError / ENOMEM (Runtime error/out of memory)
    P3: Timeout no output (infinite loop/crash)
  → Hardware detection (must be done before software repair):
    For I2C devices → mpremote exec i2c.scan()
    ├─ Device found → Software issue, enter Step 3 hierarchical decision
    └─ Empty → software I2C → lower speed → still empty → Hardware issue → Output troubleshooting guide, do not fix code

Step 3: Hierarchical decision (autofix's unique capability—decides which upstream skill to fall back to)

  Error root cause layer          →  Delegated skill         →  Repair method
  ─────────────────────────────────────────────────────
  P0: Variable name typo/missing import → autofix directly modifies file   →  Simple string replacement (no architecture change)
  P0: Driver API call error    → upy-generate      →  Inject error context, regenerate task code
  P1: I2C address conflict/pin error   → upy-select-hw     →  Reassign address/pin
  P1: Watchdog/kernel panic    → upy-generate      →  Modify scheduling strategy (add wdt.feed(), reduce sleep)
  P1: Out of memory            → upy-generate      →  Reduce buffer, gc.collect(), bytearray reuse
  P2: Sensor always returns anomaly    → upy-generate      →  Modify error handling/retry logic
  P3: Infinite loop/no output        → upy-generate      →  Analyze main.py execution flow, fix blocking point
  Hardware: I2C scan empty       → Output troubleshooting guide       →  Wiring/power/pull-up resistor/sensor/conflict (5 steps)
  Requirement: Missing sensor/wrong MCU   → upy-analyze       →  Inform user that requirement needs re-analysis

Step 4: Repair loop (max 3 attempts)

  attempt = 1..3:
    1. git commit save snapshot (autofix unique—rollback safety guarantee)
    2. Delegate to upstream skill for repair (autofix packages error context → upstream skill produces corrected code)
    3. Optional: upy-simulate PC-side quick verification (2-3s, saves serial flash latency)
    4. Delegate upy-deploy to re-flash and run
    5. Collect results
    ├─ PASS → Success exit, record repair history
    └─ FAIL → Re-enter Step 3 hierarchical decision (may escalate fallback level)

Step 5: 3 failures → Abort + Rollback + Notify user

  → git checkout rollback to pre-repair version
  → Output bottleneck summary: error type + 3 attempt contents + failure reason
  → Suggest manual troubleshooting direction
  → Record to logs/error_report.json (error backflow)

Step 6: Error data backflow (autofix unique—drives CI/CD continuous improvement)

  → logs/error_report.json records error type/traceback/repair attempts/final result
  → Periodic aggregation analysis → Discover skill.md knowledge blind spots
  → Feedback to SKILL.md of upy-autofix / upy-generate / upy-select-hw
```

**autofix's own code volume is minimal:** Core logic is `triage.py` (356 lines, error parsing + I2C hardware detection + git snapshot/rollback + hierarchical decision + loop control), repair and deployment are delegated to upstream skills, not re-implemented.

**Execution Location:** mpremote hardware detection → Local; LLM hierarchical decision → Server; Delegate upstream skills → Server (generate/select-hw/analyze) or Local (deploy/simulate)

### Module 7: Uncommon Hardware Support (Exception Path — upy-gen-driver)

Exception handling when no driver is found on either upypi or GitHub. Integrates PDF→MPY and Arduino→MPY paths, different input sources but consistent process: **Extract → Generate debug version → Hardware verify → Remove debug → Standardize**.

**Input Source → Unified Processing Flow:**

| Input Source | Extraction Method | Hardware Verification | Standardization |
|--------------|-------------------|-----------------------|-----------------|
| User provides PDF datasheet | `scripts/extract_pdf.py` (pymupdf) → Extract communication protocol/register table/initialization sequence | `mpremote resume run` max 10 rounds | upy-norm-driver |
| User provides Arduino/C++ code | `scripts/convert_arduino.py` → API mapping (digitalWrite→Pin, Wire→I2C, SPI→SPI) | Same as above | upy-norm-driver |
| Arduino code found on GitHub | clone → analyze → same as above | Same as above | upy-norm-driver |
| Only chip model provided | Search datasheet PDF → download → same as PDF process | Same as above | upy-norm-driver |

**Uncommon Hardware Complete Path:**
```
upy-analyze searches upypi + GitHub → No results
  ↓
User provides PDF / Arduino code / chip model
  ↓
upy-gen-driver:
  ├─ PDF → extract_pdf.py → LLM extracts protocol/registers
  ├─ Arduino → convert_arduino.py → API mapping
  ├─ LLM generates debug version driver (includes full self-check: register write-read verification, AT command echo, initialization step-by-step log)
  ├─ Hardware verification loop (max 10 rounds):
  │     mpremote resume run debug version driver → Capture output →
  │     LLM analyzes output PASS/FAIL → Edit driver → Verify again
  │     (resume run uses REPL, does not write Flash, fast iteration)
  ├─ Verification passes → Remove debug code (strip self-check logic + log points)
  ├─ upy-norm-driver standardize
  └─ Output to firmware/drivers/{chip}_driver/
  ↓
→ Return to main flow Phase 4 (upy-generate)
```

**Strong Constraints:**
- Driver must comply with all P0 rules of upy-norm-driver (dependency injection mandatory)
- Register table extracted from PDF must retain original reference (which page, which table)
- Arduino mapping cannot be mechanical translation; must understand the original code logic

**Execution Location:** PDF parsing, API mapping, LLM driver generation → Server; Hardware verification (mpremote resume run) → Local; Remove debug + Standardize → Server

## AI-Generated Embedded Code: MicroPython vs Arduino Platform Comparison

> From the perspective of AI-assisted development, analyze which platform is more suitable as the code generation target for "one-sentence hardware".

### Evaluation Dimensions

#### 1. Engineering Debugging Closed Loop

**MicroPython Wins**

- REPL interactive debugging: No need to re-flash, execute single-line code directly on the device for verification
- `mpremote` toolchain: `run`, `exec`, `fs` commands support hot-updating single files
- AI generates code → `mpremote run main.py` → See results immediately, very short closed loop
- Arduino: Every change requires full compilation (10s~60s) + flashing, long closed loop

#### 2. Automated Testing Framework

**MicroPython Wins**

- `unittest` module can run directly on the device
- AI can generate pin mocks, peripheral stubs, structure consistent with PC-side Python testing
- Arduino has no standard testing framework, third-party libraries like `ArduinoUnit` are complex to integrate
- AI's generation quality for Python test code is much higher than for C++ test code (training data advantage)

#### 3. Developer Perception (Compute Power/Frame Rate/Peripheral Scheduling)

**Arduino Slightly Wins, but Gap is Narrowing**

- Arduino: `micros()`/`millis()` high precision, bare-metal scheduling transparent, resource usage can be precisely calculated
- MicroPython: GC pauses, interpreter overhead not transparent, but:
  - `utime.ticks_us()` can measure frame rate
  - `micropython.mem_info()` can check memory
  - AI can generate standardized profiling templates to bridge the perception gap

#### 4. Basic Development Skills (Memory/Watchdog/Reconnection)

| Capability | MicroPython | Arduino |
|------------|-------------|---------|
| Memory Management | `gc.collect()` explicit call, easy for AI to generate | Manual malloc/free, easy for AI to make errors |
| Watchdog | `machine.WDT()` one-liner | Register operations, large platform differences |
| Reconnection Mechanism | `try/except` clear structure, high AI generation quality | State machine approach, AI easily misses edge cases |
| Exception Protection | Python exception system complete | Requires manual error codes, AI easily inconsistent |

MicroPython's exception/reconnection code has higher reliability when generated by AI.

#### 5. Generating Flowcharts/Architecture Diagrams/Dependency Graphs

**MicroPython Wins**

- Python module structure (`import`, classes, functions) is friendly for static analysis
- AI can directly extract call graphs and dependency graphs from `.py` files
- Arduino's implicit function declarations in `.ino` files and multi-file concatenation mechanism make dependency analysis difficult
- Mermaid/PlantUML generation: Python code structure is more linear, AI conversion accuracy is higher

### Conclusion

| Dimension | Winner |
|-----------|--------|
| Debugging Closed Loop | MicroPython |
| Automated Testing | MicroPython |
| Runtime Perception | Arduino (slight) |
| Basic Development Skills | MicroPython |
| Architecture Visualization | MicroPython |

**For AI-assisted embedded development, MicroPython is the more suitable target platform.** The core reason is that Python's structured nature allows AI to generate, test, and analyze code with systematically higher quality and reliability than C/C++, not just a matter of "how many libraries there are".

### upypi API Usage Specification

upypi is a structured driver package repository, call the API directly, no local vector database needed.

```
# Search for driver package
GET https://upypi.net/api/search?q={keyword}
→ { query, results: [{name, url}] }

# Get package details (compatibility + file list)
GET https://upypi.net/pkgs/{name}/{version}/package.json
→ { name, version, chips, fw, urls }

# Read driver documentation (standardized README)
GET https://upypi.net/pkgs/{name}/{version}/README.md

# Install to device
mpremote mip install https://upypi.net/pkgs/{name}/{version}/package.json
```

│   ├── {chip}.py

### User Error Data Backflow Mechanism

**Core Value:** User real failure cases are the most direct signal for discovering knowledge blind spots, driving continuous iteration of skill.md, error-patterns, and board-mapping.

```
User reports error → /fix-error processes
    ↓
Server instrumentation records (optional reporting, user authorization):
  ├── Error type + original traceback
  ├── Development board model + upypi package name/version
  ├── AI repair success attempt number / final failure
  └── Current skill.md version number used
```

**Data Analysis Flow (Weekly/Monthly Periodic):**

```
Aggregate error data
    ↓
Classify by error type:
  ├── ImportError     → upypi missing package / package name inconsistency
  ├── OSError         → Wiring error / board-mapping incorrect
  ├── AttributeError  → API usage inconsistent with README
  └── 3 repair failures     → skill.md knowledge blind spot (highest priority)
    ↓
LLM aggregate analysis → Output improvement suggestion report
    ↓
Manual review and confirmation
    ↓
Update skill.md / error-patterns / board-mapping
    ↓
Same CI/CD flow commit → Server hot reload
```

**Error Signal to Knowledge Base Mapping:**

| Error Type | Description | Update Target |
|------------|-------------|---------------|
| `ImportError` high frequency | upypi lacks a certain type of driver | Notify upypi to supplement package / upy-select-hw |
| `OSError: [Errno 19]` | Pin mapping error | board-mapping database |
| `AttributeError` | README API example inconsistent with actual | upy-generate / upy-norm-driver |
| All 3 repairs failed | Current autofix skill knowledge blind spot | upy-autofix SKILL.md |
| Specific board high-frequency errors | Board mapping or compatibility issue | board-mapping + upy-wiring |

**Privacy Principles:**
- Only collect error types and technical context, not user business code
- Clearly prompt within the plugin, only report after user checks "Help improve the product"
- Data anonymized, not linked to user identity
