# upy-publish Interface Definition

> Status: ✅ Finalized
>
> Closing Phase — Driver specification packaging + publishing to upypi. Organize the production driver generated by gen-driver into a standard directory structure, generate README/package.json/LICENSE, and optionally publish to upypi.

---

## I. Skill Overview

| Item | Content |
|------|---------|
| Phase | publish |
| Upstream Skill | upy-gen-driver (automatically entered when user selects "Publish" in Step 9) or triggered manually by user |
| Downstream Skill | None (endpoint) |
| One-line Responsibility | Read production driver → LLM generates README + package.json → Script packages standard directory → Optionally upload to upypi |

**Core Constraints:**
- No re-normalization — driver files have already been processed by gen-driver Step 5 P0 rules
- package.json deps query to upypi requires plugin-side curl execution
- Uploading to upypi is an optional step requiring user confirmation

---

## II. Plugin Input → Skill (P→S)

```json
{
  "type": "start_phase",
  "phase": "publish",
  "session_id": "uuid-xxx",
  "payload": {
    "manifest": { },
    "driver_path": "firmware/drivers/sht30_driver/sht30.py",
    "driver_dir": "firmware/drivers/sht30_driver",
    "test_passed": true,
    "chip": "SHT30"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `manifest` | object? | No | Project manifest; if present, write to package.json project field |
| `driver_path` | string | Yes | Production driver .py file path |
| `driver_dir` | string | Yes | Driver directory path (contains test_{chip}.py / wiring_{chip}.md) |
| `test_passed` | bool | Yes | Whether hardware verification passed (adds ⚠️ annotation in README when false) |
| `chip` | string | Yes | Chip model |

## III. Skill Output → Plugin (S→P)

### Message Sequence

```
Step 1: Read driver file
  → file_operation(read) → {driver_path}
  → (if test_{chip}.py exists) → file_operation(read) → test_{chip}.py
  → status_update "Analyzing driver structure..."

Step 2: Generate README
  → LLM analyzes per gen-readme template → file_operation(write) → {driver_dir}/README.md
  → status_update "✓ README.md generated"

Step 3: Query upypi dependencies
  → (Parse driver imports, extract third-party dependencies)
  → script_run(curl upypi query each dependency)
  → status_update "Querying upypi dependencies..."

Step 4: Generate package.json
  → LLM generates complete package.json → file_operation(write) → {driver_dir}/package.json
  → status_update "✓ package.json generated"

Step 5: Package standard directory
  → script_run(pack_driver.py --input {driver_dir} --output {project_dir}/{chip}_driver/)
  → status_update "✓ Standard directory generated"

Step 6: Upload to upypi (optional)
  → approval_request: publish_confirm card
  → (Confirmed) → script_run(publish_to_upypi.py --package {chip}_driver/)
  → status_update "✓ Published to upypi"
  → (Cancelled) → Skip

Output
  → phase_complete(file_list + package_info)
```

## IV. Script Changes

### pack_driver.py (New)

**Path:** `G:\MicroPython_Skills\upy-pack-driver\scripts\pack_driver.py` (copied by scaffold to `{project}/.upy/scripts/pack_driver.py`)

**Function:** Organize files in the driver directory into the upypi standard directory structure:

```
{chip}_driver/
├── code/
│   ├── {chip}.py          ← Production driver
│   └── test_{chip}.py     ← Standalone test script (if exists)
├── package.json           ← Package configuration
├── README.md              ← Usage instructions
└── LICENSE                ← MIT License
```

| Parameter | Description |
|-----------|-------------|
| `--input` | Driver directory path |
| `--output` | Output directory path |
| `--json-summary` | stdout output `{"status":"ok","package_path":"...","files":N}` |

### upy-norm-pkg / upy-gen-readme / upy-gen-pkg

**No changes needed.** The publish skill has the LLM directly generate README and package.json (pure code generation), without nesting calls to other skills. pack_driver.py handles directory organization.

## VII. Standalone Test Scenarios

### Plugin-side Testing (No Server)

1. Manually send approval_request `publish_confirm` → Verify binary choice + click confirm
2. Manually send `status_update` sequence → Verify 6-stage progress
3. Manually send `phase_complete` (file_list + table) → Verify packaged file list + package info panel

### Skill-side Testing (No Plugin)

1. Prepare normalized driver files → mock file_operation(read) returns content
2. Verify LLM correctly generates README (hardware requirements/wiring table/API/example code)
3. Verify LLM correctly generates package.json (name/version/urls/deps/chips/fw)
4. Mock upypi query response → Verify deps field is correct
5. Mock pack_driver.py → Verify standard directory structure output
6. Check all sent message JSON conforms to 02-protocol.md Schema
