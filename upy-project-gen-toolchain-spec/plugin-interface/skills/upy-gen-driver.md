# upy-gen-driver Interface Definition

> Status: ✅ Finalized
>
> Exception path — niche hardware driver generation. Triggered when no driver is found on upypi or GitHub. Generates a standardized MicroPython driver from a PDF datasheet/Arduino code/chip photo/GitHub URL/chip model.

---

## I. Skill Overview

| Item | Content |
|------|---------|
| Phase | gen-driver |
| Upstream Skill | upy-analyze (called when no driver is found), upy-autofix (called when diagnosed as missing driver), or user-triggered directly |
| Downstream Skill | upy-generate (continues the main flow using the generated driver) |
| One-line Responsibility | Input source material → Preprocessing → LLM generates debug version → Hardware verification loop (≤10 rounds) → Strip debug → P0 normalization → Generate standalone verification materials → Optional power-on test → Production driver |

**Core Constraints:**
- Step 3 hardware verification is a hard constraint; do NOT proceed to Step 4/5 before SELF_TEST_PASS
- If no device is connected, pause and wait for user confirmation; do not skip with the reason "no device currently available"
- `extract_pdf.py` / `convert_arduino.py` only perform extraction + mapping; understanding is done by the LLM
- Arduino translation must not be mechanical line-by-line; the original logic must be understood and rewritten using idiomatic MicroPython
- Dependency injection is a hard requirement: I2C/SPI/UART instances must be passed in externally
- No infinite polling: all wait/polling loops must have a timeout

---

## II. Plugin Input → Skill (P→S)

```json
{
  "type": "start_phase",
  "phase": "gen-driver",
  "session_id": "uuid-xxx",
  "payload": {
    "manifest": { },
    "source": null
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `manifest` | object? | No | Existing manifest (from upstream analyze), null if none |
| `source` | object? | No | Input material pre-filled by the plugin. Triggers `approval_request` for user upload when null. |

### source Structure (when pre-filled by plugin)

```json
{
  "source": {
    "type": "pdf",
    "files": [
      {
        "name": "sht30_datasheet.pdf",
        "mime_type": "application/pdf",
        "size": 245760,
        "local_path": "/tmp/uploads/sht30_datasheet.pdf",
        "content": "(extracted text content...)",
        "thumbnail_base64": "data:image/png;base64,iVBOR..."
      },
      {
        "name": "chip_silkscreen.jpg",
        "mime_type": "image/jpeg",
        "size": 98304,
        "content_base64": "/9j/4AAQSkZJRg...",
        "thumbnail_base64": "data:image/jpeg;base64,/9j/4AAQ..."
      }
    ],
    "url": "https://github.com/adafruit/Adafruit_SHT31",
    "chip_model": "SHT30"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | Yes | Input type: `pdf` / `arduino` / `image` / `github_url` / `chip_model` |
| `files` | array? | Depends on type | List of PDF/Arduino/image source files. Max 5 files, each ≤10MB |
| `files[].name` | string | Yes | Original filename |
| `files[].mime_type` | string | Yes | MIME type |
| `files[].size` | int | Yes | File size (bytes) |
| `files[].local_path` | string | No | Local path on the plugin side (for `script_run` reference) |
| `files[].content` | string? | No | PDF/Arduino type: preprocessed text content (plugin executes extract_pdf/convert_arduino in advance) |
| `files[].content_base64` | string? | No | Image type: base64 encoded raw image (for server LLM multimodal recognition) |
| `files[].thumbnail_base64` | string? | No | data URI thumbnail (150×150), generated by the plugin |
| `url` | string? | No | GitHub repository URL, required when type=github_url |
| `chip_model` | string? | No | Only chip model, required when type=chip_model |

**Plugin-side Preprocessing Rules:**
- PDF: Plugin executes `extract_pdf.py` first, fills text into `content`. If not preprocessed (content is empty), server sends `script_run`
- Arduino: Plugin executes `convert_arduino.py` first, fills mapping result into `content`. If not preprocessed, server sends `script_run`
- Image: Send `content_base64`, no preprocessing; server LLM multimodal recognition identifies chip model/silkscreen
- Thumbnail: Plugin generates 150×150. PDF → render page 1, Image → scale, Code → file type icon + filename

## III. Skill Output → Plugin (S→P)

### Message Sequence

```
Step 0: Input Judgment (when source=null or source.files is empty)
  → approval_request: gen_driver_input card

Step 1: Preprocessing (branches based on source.type)
  ├─ pdf:
  │   (if content is filled → skip script, go directly to LLM analysis)
  │   (if content is empty) → script_run(extract_pdf.py --input {local_path} --output docs/driver_extracted.json --json-summary)
  │   → file_operation(read) → docs/driver_extracted.json
  │   → status_update "Analyzing datasheet..."
  │   → LLM generates understanding.json → file_operation(write) → docs/driver_understanding.json
  │   → status_update "Chip: {chip}, Protocol: {I2C/SPI/UART}, Address: 0xXX, ID Register: {Yes/No}"
  │
  ├─ arduino:
  │   (if content is filled → skip script)
  │   (if content is empty) → script_run(convert_arduino.py --input {local_path} --output docs/driver_mapping.json --json-summary)
  │   → file_operation(read) → docs/driver_mapping.json
  │   → LLM reads both original source code + mapping JSON → understands logic → translates to MicroPython
  │
  ├─ image:
  │   → (content_base64 already in payload) → LLM multimodal recognition identifies chip model/silkscreen
  │   → Chip identified → WebSearch datasheet → (find PDF URL → approval_request notifies user to download → go back to pdf path)
  │   → Cannot identify → approval_request informs user, requests supplementary information
  │
  ├─ github_url:
  │   → WebFetch pulls repository source code
  │   → status_update "Pulling GitHub source code..."
  │   → (Pull successful → LLM analyzes source structure → translates to MicroPython)
  │   → (Pull failed → approval_request informs user, requests manual code provision)
  │
  └─ chip_model:
      → WebSearch "{chip_model} datasheet PDF"
      → (Find PDF URL) → approval_request notifies user to download PDF → go back to pdf path
      → (Cannot find) → approval_request informs user, requests manual upload

Step 2: Generate Debug Driver
  → status_update "Generating debug driver..."
  → file_operation(write) → firmware/drivers/{chip}_driver/{chip}_debug.py
  → status_update "✓ Debug driver generated (contains N self-tests)"

Step 3: Hardware Verification Loop (≤10 rounds)
  → device_command(devs)
  → (No device) → approval_request: gen_driver_no_device card
     ├─ Re-detect → go back to device_command(devs)
     ├─ Skip → go to Step 4 (marked ⚠️ Not hardware verified)
     └─ Later → phase_complete(result=partial), can continue anytime
  → (Device found) → Enter loop:
     ┌─ script_run(run_on_device.py --com {COM} --file {debug_py} --capture)
     ├─ file_operation(read) → logs/driver_verify_round{N}.log
     ├─ LLM analyzes output
     ├─ status_update for each round result (PASS/FAIL + error summary)
     ├─ SELF_TEST_PASS → Exit loop, go to Step 4
     ├─ Failure → Edit + file_operation(write) → next round
     └─ 10 rounds exhausted → phase_complete(result=partial, errors)

Step 4: Strip Debug → Production Driver
  → (Precondition check: Step 3 executed + SELF_TEST_PASS + debug version file exists)
  → (Condition not met → go back to Step 3)
  → status_update "Generating production driver..."
  → file_operation(write) → firmware/drivers/{chip}_driver/{chip}.py
  → status_update "✓ Production driver generated"

Step 5: Normalization
  → file_operation(read) → {chip}.py
  → (LLM rewrites according to upy-norm-driver P0 rules one by one)
  → (Middleware library skips #16/#16a/#16b/#16c/#29/#34-#38)
  → file_operation(write) → {chip}.py (overwrite)
  → status_update "✓ Normalization complete (N P0 rules executed)"

Step 6: Generate Standalone Verification Materials (always present, regardless of how gen-driver was triggered)
  → (LLM extracts pin/bus/address/power requirements from understanding.json)
  → file_operation(write) → firmware/drivers/{chip}_driver/test_{chip}.py
     (Minimal script: import driver → init → read data once → print → deinit, does not depend on manifest/other modules)
  → file_operation(write) → firmware/drivers/{chip}_driver/wiring_{chip}.md
     (Single chip wiring table: chip pin → MCU pin + power requirements + pull-up/current limiting reminders)
  → status_update "✓ Standalone test script + wiring reference generated"

Step 7: Standalone Power-On Test (approval_request, optional)
  → approval_request: gen_driver_standalone_test card
     (Displays wiring table + power requirements + pull-up resistor reminders)
     ├─ "Wired, start test" → script_run(run_on_device.py --file test_{chip}.py --capture)
     │   → file_operation(read) → logs/driver_standalone_test.log
     │   → PASS → status_update "✓ {chip} standalone test passed"
     │   → FAIL → status_update "✗ {chip} standalone test failed" → go back to Step 3 for troubleshooting
     └─ "Manual test later" → Skip, files already generated, testable anytime

Step 8: Update Manifest (conditional execution)
  → (Only executed when manifest exists — gen-driver called from analyze/autofix)
  → (Skip when triggered independently)
  → file_operation(read) → project-manifest.json
  → (Server modifies drivers field, appends new driver)
  → file_operation(write) → project-manifest.json

Step 9: Next Step Selection (always present)
  → approval_request: gen_driver_next_step card
     ├─ "Standardize package + publish to upypi" → phase_complete(next_phase="publish")
     ├─ "Integrate into project and continue development" → phase_complete(next_phase="generate")
     └─ "Finish" → phase_complete(next_phase=null)

Output
  → phase_complete(file_list + diagnostics table + next_phase)
```

### script_run — run_on_device.py (Step 7 Standalone Test)

```json
{
  "type": "script_run",
  "payload": {
    "script_id": "standalone_test",
    "interpreter": "python",
    "script": ".upy/scripts/run_on_device.py",
    "args": ["--com", "{COM}", "--file", "firmware/drivers/{chip}_driver/test_{chip}.py", "--capture", "--timeout-ms", "15000"],

```

**Shares `run_on_device.py` with Step 3, only `--file` and `--timeout-ms` differ.** The standalone test script is simpler (single data read), timeout only needs 15s.
