# upy-generate Interface Definition

> Status: ✅ Finalized
>
> Phase 4 — Business code generation (the heaviest skill). Downloads drivers, generates DI-architecture business code, Mock layer, and unit tests. Supports both full generation and targeted fixes triggered by autofix.

---

## I. Skill Overview

| Item | Content |
|------|---------|
| Phase | generate |
| Upstream Skill | upy-scaffold (full mode) / upy-autofix (fix mode) |
| Downstream Skill | upy-simulate (manual trigger) / upy-deploy (automatic entry) |
| One-line Responsibility | Download drivers → Understand API → Generate factory + Mock → Generate tasks → Supplement config → DI assembly → Test generation → Multi-layer validation |

**Core Constraints:** No code except main.py imports `machine`. DI duck typing. `print()` + logger dual-write. Each task has its own try/except.

**Two Operation Modes:**

| Mode | Trigger | Behavior |
|------|---------|----------|
| `full` | upy-scaffold completes | Full 9-phase flow, from driver download to test generation |
| `fix` | upy-autofix delegates | Read only the error file → Minimal modification → Lint validation → Return diff |

## II. Plugin Input → Skill (P→S)

### full mode

```json
{
  "type": "start_phase",
  "phase": "generate",
  "session_id": "uuid-xxx",
  "payload": {
    "mode": "full",
    "manifest": { "...complete project-manifest.json (phase: scaffold)..." }
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `mode` | string | Yes | `"full"` |
| `manifest` | object | Yes | Complete manifest, must contain `scaffold_mode` + `devices[].driver` + `pinout` |

### fix mode (called by upy-autofix)

```json
{
  "type": "start_phase",
  "phase": "generate",
  "session_id": "uuid-xxx",
  "payload": {
    "mode": "fix",
    "manifest": { "...complete manifest..." },
    "error_context": {
      "traceback": "Traceback (most recent call last):\n  File \"tasks/sensor.py\", line 42, in sensor_read\n    data['temp'] = sensor.measure()[0]\nTypeError: 'NoneType' object is not callable\n",
      "file_path": "firmware/tasks/sensor.py",
      "line_number": 42,
      "driver_name": "SHT30",
      "error_type": "P0_driver_api",
      "attempt_number": 2,
      "previous_attempts": [
        {
          "attempt": 1,
          "strategy": "Added None check to sensor.measure() return value",
          "files_changed": ["firmware/tasks/sensor.py"],
          "result": "Same error, ineffective"
        }
      ]
    }
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `mode` | string | Yes | `"fix"` |
| `manifest` | object | Yes | Complete manifest |
| `error_context.traceback` | string | Yes | Original Python traceback |
| `error_context.file_path` | string | Yes | Path to the error file |
| `error_context.line_number` | number | Yes | Error line number |
| `error_context.driver_name` | string | No | Involved driver name |
| `error_context.error_type` | string | Yes | autofix classification: `P0_typo_import` / `P0_driver_api` / `P1_pin_addr` / `P1_wdt_mem` / `P2_sensor` / `P3_loop_no_output` / `unknown` |
| `error_context.attempt_number` | number | Yes | Current attempt number (1~3) |
| `error_context.previous_attempts` | array | No | Strategies and results of previous attempts |

## III. Skill Output → Plugin (S→P)

### full mode message sequence

```
Phase 1: Download drivers
  → status_update "Searching and downloading drivers..."
  → [Server internal] Run download_drivers.py (stdin manifest → stdout JSON)
  → [Server internal] LLM receives JSON, obtains all driver source code content
  → file_operation(write) × N writes driver files to plugin-side firmware/lib/
  → status_update "✓ SSD1306 → upypi (ssd1306-driver v1.3.0)"
  → status_update "⚠ SHT30 → Driver not found"
  → status_update "✓ 2/3 drivers downloaded, 6 files total"

Phase 2: Understand driver → Generate factory + Mock
  → status_update "Reading SSD1306 driver source code..."
  → [Server internal] LLM reads driver source code from download_drivers.py output JSON
  → [Server internal] Generate drivers/ssd1306_driver/__init__.py (factory function + scan function)
  → [Server internal] Generate drivers/ssd1306_driver/mock.py (Mock class)
  → file_operation(write) × 2 writes to plugin side
  → status_update "✓ SSD1306 factory + Mock generated"
  → (Repeat for each device)
  → status_update "✓ 3 devices completed (factory + Mock)"

Phase 3: Generate task files
  → status_update "Generating sensor_task.py..."
  → [Server internal] LLM generates task code based on manifest.requirements + devices
  → file_operation(write) × M
  → status_update "✓ sensor_task.py / display_task.py / alarm_task.py generated"

Phase 4: Supplement conf.py
  → file_operation(read) gets current conf.py (version generated by scaffold)
  → [Server internal] LLM supplements business constants (thresholds, calibration values, etc.)
  → file_operation(write) overwrites conf.py
  → status_update "✓ conf.py supplemented with business constants"

Phase 5: Generate main.py (DI assembly)
  → file_operation(read) gets current main.py (skeleton generated by scaffold)
  → [Server internal] LLM fills in DI assembly chain + task registration + log initialization
  → file_operation(write) overwrites main.py
  → status_update "✓ main.py DI assembly complete"

Phase 6: Generate test files
  → [Server internal] LLM generates PC-side unit tests + device-side smoke tests
  → file_operation(write) × 2~4
  → status_update "✓ test/pc/test_sensor.py / test/device/test_smoke.py"

Phase 7: Multi-layer validation
  → script_run(flake8) → Fail → [Internal] LLM fixes → Re-run → Pass
  → script_run(pylint) → Same as above
  → script_run(check_mpy_imports.py) → Same as above
  → script_run(check_dead_config.py) → Same as above
  → script_run(check_skeleton_compliance.py) → Same as above
  → status_update "✓ All 5 validations passed"

Phase 8: Final review
  → [Server internal] LLM final review (driver API correctness, requirement coverage, test coverage, import compatibility)

Phase 9: Output
  → phase_complete (file tree + device status table + manifest_content)
```

### fix mode message sequence

```
Step 1: Read current file
  → file_operation(read) × 1~3 (read error file + related dependency files)

Step 2: Analyze fix
  → [Server internal] LLM reads traceback + current code + historical attempts → designs fix strategy

Step 3: Write fix
  → file_operation(write) × 1~2 (write only modified files)
  → status_update "Fixing tasks/sensor.py line 42..."

Step 4: Validate
  → script_run(flake8) → script_run(pylint)
  → status_update "✓ Lint validation passed"

Step 5: Return
  → phase_complete (code_diff + fix summary)
```

**fix mode does not send approval_request.** The user is in the autofix flow; fixes are executed silently.

**fix mode must read first.** The file may have been modified by a previous fix attempt; it cannot rely on memory from the initial full generation.

#### script_run validation chain

5 validation scripts execute sequentially. Any step fails → LLM reads errors list → fixes → re-runs that script. Each script has a 15-second timeout.

```json
{
  "type": "script_run",
  "payload": {
    "script_id": "gen_lint_001",
    "interpreter": "python",
    "script": "check_skeleton_compliance.py",
    "args": ["--manifest", "-", "--project-dir", "{project_dir}"],
    "cwd": "{skill_dir}",
    "timeout_ms": 15000
  }
}
```

All validation scripts share a unified output format:

```json
{
  "status": "pass",
  "errors": [
    {
      "file": "firmware/tasks/sensor.py",
      "line": 5,
      "message": "import logging is not in the MicroPython whitelist",
      "severity": "error"
    }
  ],
  "warnings": [],
  "summary": "1 error, 0 warnings"
}
```

| Script | Check Content | Exit Code |
|--------|---------------|-----------|
| `flake8` | Code style + syntax errors | Non-zero = issues |
| `pylint` | Code quality (uses .pylintrc config) | Non-zero = issues |
| `check_mpy_imports.py` | Scans all imports, checks against 63 MPY whitelist | Non-zero = violations |
| `check_dead_config.py` | Checks if constants defined in conf.py are referenced | Non-zero = dead config |
| `check_skeleton_compliance.py` | 5 skeleton compliance checks | Non-zero = violations |

#### file_operation — driver download write

`download_drivers.py` runs on the server side (using server network), outputs JSON to stdout. Server parses it:

```json
{
  "type": "file_operation",
  "payload": {
    "op_id": "gen_dl_001",
    "op": "write",
    "path": "firmware/lib/ssd1306.py",
    "content": "# MicroPython SSD1306 OLED driver...\n...",
    "encoding": "utf-8"
  }
}
```

At the same time, the LLM directly obtains the driver source code content from the JSON to enter Phase 2, without needing an additional `file_operation(read)`.

A total of 11 changes:

| # | Location | Current Behavior | Change To | Reason |
|---|----------|-----------------|-----------|--------|

### 5.2 check_mpy_imports.py (New)

**Path:** `G:\MicroPython_Skills\upy-generate\scripts\check_mpy_imports.py`

Replaces the current Phase 7C `find` + `grep` manual flow.

**Check logic:**
1. Scan all `.py` files under `firmware/` (excluding `firmware/lib/`)
2. Extract top-level module name X from `import X` and `from X import Y`
3. Check against the 63 MicroPython whitelist
4. Not in whitelist but has a `.py` file with the same name under `firmware/` → report error: should use relative import
5. Not in whitelist and no file under `firmware/` → report warning: may not be supported

```bash
python check_mpy_imports.py --project-dir {project_dir}
# stdout → {"status": "pass", "errors": [], "warnings": [], "summary": "..."}
# exit: 0=pass, 1=fail
```

### 5.3 check_dead_config.py (New)

**Path:** `G:\MicroPython_Skills\upy-generate\scripts\check_dead_config.py`

Replaces the current Phase 7D inline Python + grep.

**Check logic:**
1. Extract all `UPPER_CASE = value` constant names from `conf.py`
2. Scan all `.py` files under `firmware/` + `test/` (excluding `conf.py` itself and `firmware/lib/`)
3. Check if each constant is referenced (`from conf import X` or `conf.X`)
4. Unreferenced → dead config, report warning

```bash
python check_dead_config.py --project-dir {project_dir}
# stdout → {"status": "pass", "errors": [], "warnings": [...], "summary": "..."}
```

### 5.4 check_skeleton_compliance.py (New)

**Path:** `G:\MicroPython_Skills\upy-generate\scripts\check_skeleton_compliance.py`

Ensures that code generated by upy-generate adheres to the skeleton set by upy-scaffold.

**Five checks:**

| # | Check Item | Rule |
|---|------------|------|
| 1 | Scheduler mode consistency | The scheduling method (Timer/asyncio/_thread) in main.py matches `manifest.scaffold_mode` |
| 2 | board.py pin references | GPIO numbers must come from `from board import` rather than hardcoded `Pin(4)` (except main.py) |
| 3 | conf.py constant references | Thresholds/intervals etc. must `from conf import` rather than hardcoded (except main.py) |
| 4 | DI chain completeness | Driver creation in main.py must go through the `drivers/*/__init__.py` factory function |
| 5 | No direct import of machine | `tasks/*.py` must not contain `import machine` or `from machine import` |

```bash
python check_skeleton_compliance.py --project-dir {project_dir} --manifest - < manifest.json
# stdout → {"status": "pass", "errors": [], "warnings": [], "summary": "..."}
```

### 5.5 Two-stage file_operation(read) (fix mode specific)

In fix mode, the LLM needs to first read the current content of the error file. If the error involves multiple files (e.g., sensor.py calls a function whose signature has changed), multiple files need to be read.

The LLM decides which files to read, but it **must** read at least the file pointed to by `error_context.file_path`.

## VII. Independent Test Scenarios

### Plugin-side Tests (No Server)

1. Manually send a status_update sequence (20 messages) → Verify long timeline scrolling and collapsing
2. Manually construct phase_complete (with file_tree + table) → Verify file tree + table
3. Manually construct phase_complete (fix mode, with code_diff) → **Focus on testing diff view**:
   - Single line modification
   - Multi-line modification
   - New code block
4. Manually send a script_run × 5 sequence → Verify validation panel updates one by one

### Skill-side Tests (No Plugin)

1. **Full mode complete flow:**
   - mock_plugin sends start_phase (mode=full, manifest=temp/humidity project, phase: scaffold)
   - Auto-verify: download_drivers.py stdout JSON schema is correct
   - Auto-verify: Generated factory/Mock method signatures match driver source code
   - Auto-verify: All 5 validation scripts pass
   - Check that all file_operation content is valid Python
2. **Fix mode:**
   - First generate a complete project via full mode
   - mock_plugin sends start_phase (mode=fix, error_context={traceback, file_path, ...})
   - Verify that file_operation(read) was sent first
   - Verify that only file_operation(write) for the modified file was sent, no other files changed
   - Verify that phase_complete contains the code_diff artifact
3. **Fix mode repeated calls (simulating autofix 3 re-entries):**
   - First fix → modify sensor.py
   - Second fix (same error) → first read sensor.py → find existing None check → change strategy → modify main.py
   - Third fix → Verify LLM correctly uses previous_attempts information
4. **download_drivers.py newline fix:**
   - Construct a fake driver file containing `\r\r\n` (mock HTTP response)
   - Verify that Python compile passes after download
5. **Validation script boundary cases:**
   - check_mpy_imports.py: File containing `import typing` → Expected fail
   - check_dead_config.py: conf.py defines 5 constants, only 3 used → Expected fail
   - check_skeleton_compliance.py: tasks/sensor.py hardcodes GPIO4 → Expected fail
