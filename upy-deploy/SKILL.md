# Device Deployment and Run Skill

## Role

Given a `project-manifest.json` (phase: generate), upload `firmware/` to the MicroPython device, run it, capture output, and perform a preliminary assessment. **Do not fix errors (that is the responsibility of upy-autofix).**

---

## Pre-checks

```bash
python --version
mpremote --version
```

Dependencies: Python 3 standard library + mpremote.

---

## Core Design

### 3-second Startup Delay in main.py

upy-generate Phase 5 constraint #0 requires that the first line of code in main.py is `time.sleep(3)`. This gives deploy enough time to reconnect after a soft reset:

```
mpremote soft-reset
  → Device restarts, USB disconnects
  → Windows re-enumerates COM (1~3s)
  → mpremote reconnects + resumes
  → main.py is exactly within the 3s delay window, so no output is lost upon connection
```

Without this delay, main.py might have already run halfway (log headers, I2C scan results lost) by the time mpremote connects.

### Dual Channel: print() + Rotating Logs

deploy obtains device output from two sources:

| Source | Method | Characteristics |
|------|------|------|
| REPL real-time output | mpremote persistent session | `print()` visible immediately, but lost if REPL disconnects |
| Device-side log files | `mpremote fs cp :log/run_*.log` | Persist across reboots, readable even if REPL disconnects |

deploy Phase 5 fetches log files as a supplement—even if the persistent session misses some output due to disconnection, the log files can fill the gaps.

## Execution Steps

### Phase 1: Upload Files

**Primary method**: Use the project's own `tools/flash_device.py` (generated by upy-scaffold):

```bash
# Compile .py → .mpy (excluding main.py/boot.py) + upload, --no-reset lets deploy control the reset timing
python tools/flash_device.py --compile --upload --no-reset --port <COM>
```

`flash_device.py` automatically: recursively traverses firmware/, skips .gitkeep, creates remote directories, copies files one by one, and uploads main.py last.

**Fallback method** (when `tools/flash_device.py` does not exist, use mpremote directly):

```bash
cd firmware

# 1. Create necessary directories (first check with ls if they already exist)
mpremote connect <device> resume fs ls :log >/dev/null 2>&1 || \
    mpremote connect <device> resume fs mkdir :log
mpremote connect <device> resume fs ls :lib >/dev/null 2>&1 || \
    mpremote connect <device> resume fs mkdir :lib
mpremote connect <device> resume fs ls :drivers >/dev/null 2>&1 || \
    mpremote connect <device> resume fs mkdir :drivers
mpremote connect <device> resume fs ls :tasks >/dev/null 2>&1 || \
    mpremote connect <device> resume fs mkdir :tasks

# 2. Upload lib/ and drivers/ (dependencies first, entry point last)
for f in lib/**/*.py; do
    mpremote connect <device> resume fs cp "$f" ":$f"
    sleep 0.5
done
for f in drivers/**/*.py; do
    mpremote connect <device> resume fs cp "$f" ":$f"
    sleep 0.5
done

# 3. Upload configuration and task files
mpremote connect <device> resume fs cp board.py :board.py
mpremote connect <device> resume fs cp conf.py :conf.py
mpremote connect <device> resume fs cp boot.py :boot.py
for f in tasks/*.py; do
    mpremote connect <device> resume fs cp "$f" ":$f"
    sleep 0.5
done

# 4. Upload main.py last (once it exists, the device will execute it on startup)
mpremote connect <device> resume fs cp main.py :main.py
```

**Upload order**: lib/ → drivers/ → config → tasks → main.py (last). `flash_device.py` ensures this order through sorting.

**Execute from the `firmware/` directory**: After `cd firmware`, all paths are relative, and remote paths (`:` prefix) match local paths, avoiding an extra `firmware/` directory prefix.

**Regarding `resume`**: In the fallback method, `fs cp` must include `resume`. `flash_device.py` operates during the upload phase when the device is idle; testing shows it works without `resume`.

### Phase 2: Verify File Integrity

```bash
# List the device file tree
mpremote connect <device> resume fs tree

# Or compare directory by directory
mpremote connect <device> resume fs ls :lib/
mpremote connect <device> resume fs ls :drivers/
```

Compare against the local file list; retransmit any missing files.

---

### Phase 3: Soft Reset + Wait for Reconnection

```bash
# Do not use resume — we want the device to restart
mpremote connect <device> soft-reset
```

Call `wait_for_device(com, timeout=60)` — poll until the device responds:

```
Every 2 seconds:
  mpremote connect <com> resume exec "print(1)"
  → Success (received "1"): Device ready, main.py is within the 3s delay window
  → Failure (connection error/no response): Continue waiting
  → Timeout after 60s: Report FAIL, terminate
```

**Note**: After `soft-reset`, the device restarts, and Windows may change the COM port. If `connect <com>` consistently fails, use `mpremote connect list` to rescan for COM ports.

---

### Phase 4: Persistent Session + Capture Output

The device automatically runs main.py. Establish a persistent session to capture the complete output (60s timeout).

**Windows (subprocess + pipe)**:

```python
import subprocess, threading, time

proc = subprocess.Popen(
    ["mpremote", "connect", com, "resume", "repl"],
    stdin=subprocess.PIPE, stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT, bufsize=0,
)

output_lines = []
def reader():
    for line in iter(proc.stdout.readline, b""):
        text = line.decode("utf-8", errors="replace")
        output_lines.append(text)
        print(text, end="")  # Real-time echo

t = threading.Thread(target=reader, daemon=True)
t.start()

# Wait 60s or for the scheduler stable running flag
deadline = time.time() + 60
while time.time() < deadline:
    if any("starting scheduler" in l.lower() for l in output_lines):
        break  # Scheduler started = success, end early
    time.sleep(1)

proc.terminate()
```

**Linux/macOS (PTY, see mpremote-live-session)**:

Use `pty.openpty()` + `os.fork()` to establish a real terminal session. More reliable than pipe, supports Ctrl+C forwarding.

**Timeout strategy**:
- Seeing `"starting scheduler"` or equivalent flag → early PASS, end capture
- 60s without key flags but no errors → PASS (maybe just no print flag)
- Traceback in output → FAIL

### Phase 5: Fetch Device-Side Logs

The persistent session may lose some output due to disconnection. Supplement by reading from the device file system.

**Get log directory**: Read `LOG_DIR` from `firmware/conf.py`, default is `/log`:

```bash
LOG_DIR=$(python -c "import sys; sys.path.insert(0,'firmware'); from conf import LOG_DIR; print(LOG_DIR)" 2>/dev/null || echo "/log")
```

**5a. Instant analysis (fs cat → pipe to log_report.py)**:

```bash
python tools/read_device_log.py --port <COM> --log-dir $LOG_DIR \
    | python tools/log_report.py --input > deploy_log_report.json
```

`read_device_log.py` internally uses `fs ls :{log_dir}/` to dynamically discover all `run_*.log` files, then uses `fs cat` to read each file's content and output to stdout. `log_report.py` reads from stdin and outputs structured JSON.

**5b. Download raw files (fs cp → local, for autofix)**:

```bash
python tools/read_device_log.py --port <COM> --log-dir $LOG_DIR --download ./deploy_logs/
```

`fs cp` downloads each `run_*.log` to the local `deploy_logs/` directory. These raw .log files are passed to the upy-autofix server-side LLM for deep analysis in case of FAIL.

**Fallback method** (when `tools/read_device_log.py` does not exist):

```bash
mpremote connect <device> resume fs ls :log/
mpremote connect <device> resume fs cp :log/run_0.log deploy_log_0.log
mpremote connect <device> resume fs cp :log/run_1.log deploy_log_1.log
# ... Download files one by one based on the actual list from fs ls
python tools/log_report.py deploy_log_*.log
```

Log files contain a complete record from boot to crash (even if REPL disconnects), making them a key data source for determining the cause of failure.

---

### Phase 6: Preliminary Assessment

**Purely local rules, no server-side AI calls.** Combine two data sources for the assessment:

**Data Source 1 — REPL real-time output (grep rules):**

```
Assessment rules:
  output contains "Traceback (most recent call last)" → FAIL (Python exception)
  output contains "rst cause:"                          → FAIL (Hardware reset)
  output contains "Guru Meditation Error"               → FAIL (ESP32 kernel panic)
  output contains "MemoryError" / "ENOMEM"              → FAIL (Out of memory)
  No output within 60s                                  → FAIL (Device unresponsive)
  Other                                                → PASS ✓
```

**Data Source 2 — `deploy_log_report.json` (structured output from log_report.py):**

```bash
# Check error_count in JSON
error_count=$(python -c "import json; print(json.load(open('deploy_log_report.json'))['error_count'])")
```

If `error_count > 0` and the REPL grep did not hit → still FAIL (errors in logs, but REPL output may have been lost due to disconnection). Check error levels one by one: P0 and P1 levels directly cause FAIL, P2 level serves as auxiliary information.

```
Output structure:
  { status: "PASS" | "FAIL",
    com_port: "COM3",
    output: <Complete captured REPL output>,
    log_report: <deploy_log_report.json content>,
    log_files: <List of raw .log files in deploy_logs/>,
    deployed_files: <fs tree output>,
    device_info: { firmware, flash_free, freq }
  }
```

PASS → End of process, display run results.
FAIL → Pass the complete context (REPL output + deploy_log_report.json + deploy_logs/*.log) to upy-autofix.

## Platform Differences

| | Windows | macOS | Linux |
|------|------|------|------|
| Port identifier | `COM3` | `/dev/tty.usbmodem*` | `/dev/serial/by-id/*` or `mpy-dev tty` |
| Persistent session | subprocess + pipe | PTY | PTY |
| Port change handling | Rescan with `mpremote connect list` | Path stable (USB slot unchanged) | `mpy-dev` lookup by device name |
| Reconnection wait | Yes (COM number may change) | Usually not needed | Not needed (by-id is stable) |

## Strong Constraints

- **No server-side AI calls**: deploy is a purely local operation for a fast feedback loop. Error analysis is left to autofix.
- **No code modification**: Only upload, run, and assess. Fixing is autofix's responsibility.
- **Must compile with mpy-cross before uploading**: Except for main.py and boot.py, all .py files must be compiled to .mpy before uploading. Reduces Flash usage and avoids out-of-memory errors during import on the device.
- **main.py / boot.py must remain .py**: MicroPython startup only recognizes .py entry files; compiled .mpy files cannot be executed automatically.
- **main.py must be uploaded last**: It triggers the device's main startup logic; upload dependencies first, then the entry point.
- **fs cp must include resume**: `fs cp` without `resume` will perform a soft reset first, potentially corrupting files during transfer.
- **Reconnection timeout of 60s**: Covers USB re-enumeration + main.py's 3s delay + startup initialization.
- **Dual log source capture**: Persistent session REPL output + device-side rotating log files, complementing each other.
